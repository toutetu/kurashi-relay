## 重点的に見てほしい点

### A. 意味の反転が正しく反映されているか(最重要)

このPRの本質は「**ママと決める(未決定)**」→「**ママと決めた(決定済み)**」という意味の反転です。
反転を取りこぼすと、画面に**嘘の情報**が出ます。

1. 「🎀 ママと決めた」が**選択内容を実際に保存している**か。
   旧実装のように state だけ書いて中身を保存しない経路が残っていないか。
2. `これにする!` と `ママと決めた` の保存内容が**完全に同じ**で、違いが `decided_with` **だけ**か。
   保存処理が2本に分岐していたら指摘(スペック §7-1 で一本化を要求している)。
3. 母用 `KoekakeMusumeSummaryCard.tsx` で、`decided_with === 'mama'` のときに
   **決定内容が隠れていないか**。旧実装は「ママと決める」という文言で内容を置き換えていた。
   新仕様は**内容を出したうえで 🎀 を添える**。ここが最も間違えやすい。
4. 「今は決めない」が**保存せず閉じる**ままか(挙動を変えていないか)。

### B. 導出ロジックの正しさ

5. `plan.*_state` への参照が完全に消え、スペック §3 の導出表どおりに判定しているか。
6. 「決定済み」の判定が**空配列と未設定を取り違えていないか**。
   例: `items.tomorrow_plan` が `[]` のときに「決定済み」と誤判定しないか。
7. `decided_with` をカードごとに読むとき、**行が0件のカードで落ちないか**
   (`items.x[0].decided_with` のような安全でないアクセスが無いか)。

### C. 表示条件

8. 「🔮 明日 なにする?」が**夏休みモードのみ**表示され、位置が
   「🎒 明日 何がいる?」の**直上**か。

   カードが消えること。そのとき保存済みデータが**壊れたり消えたりしない**か
   (表示されないだけで残るのが正)。
10. 「その他」自由入力が既存の `splitOtherTitle` / `msm-other-input` を再利用しているか。
