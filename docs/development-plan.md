# くらしリレー 実装計画

更新日: 2026-07-20

この文書は、くらしリレー全体の実装状況、未実装領域、開発順序を示す恒久ドキュメントである。
今後「実装計画」「開発計画」「次に何を作るか」を確認するときは、まず本書を参照する。

個別機能の詳細仕様は `docs/wip/<機能>/`、DB統合の完了保管は
`docs/archive/phases/database-unification/`、設計判断は
`docs/design-decisions.md`、データ構造の正は `docs/data-model.md` を参照する。

## 1. 現在地

くらしリレーは、毎日の記録に使えるPoCまで進んでいる。

- 娘の「くらしのおしごと」と母の「家事手帖」は、回数、取消、ポイント・ごほうびをDBへ保存できる。
- 日付別・母娘別の記録を閲覧できる。
- 母用の声かけ画面は、声かけ、完了状態、取消、再通知時刻をDBへ保存できる。
- 娘用ホームは、今日・明日の見通し、夏休みモード、「ママと決めた」、振り返りをDBへ保存できる。
- React SPAとLaravel APIは本番公開済みである。

一方、プロダクトの最終目的である「予定と実績の差を把握し、支援者との役割分担につなげる」流れは未完成である。
ホーム周辺には固定データと画面内だけの操作が残り、予定入力、支援引き継ぎ、支援者向けレポート、
Google Calendar、通知配信は未実装である。APIは家族共有トークンで保護済みである（DR-042）。

## 4. 実装の優先順位

- **Gate**: 後続Phaseへ進む前に満たす安全・仕様上の関門。
- **Phase**: 基盤を作るデプロイ可能な実装単位。
- **Product Phase**: 基盤の安全ゲート通過後に利用価値を追加する実装群。

以下は原則として記載順に実行する。完了済みのGate・Phaseは、残る運用確認だけを明記する。

### Gate 0: 本番確認を閉じる

状態: **完了（2026-07-19）**。DR-028の機能優先方針と非致命リスク受容に基づく。

完了根拠:

- 本番で主要APIの読取・書込、JST時刻、夏休み計画の母画面反映を確認し、重大な不整合を解消した。
- ローカル品質ゲートと本番スモークが通過した。詳細は
  [Gate 0 本番確認記録](archive/phases/gate0-production-verification-2026-07-19.md) に保管した。

Gate 0後も観測を続けるが、機能実装を止めない項目:

1. Laravel Cloud画面での `migrate:status` 出力保存。既存行の時刻補正から差分migrationの効果は確認済み。
2. 娘さんとの実端末、機内モード、日跨ぎ、朝・昼・夜、10回報酬の観測。
3. H1〜H3/M1〜M3の根治。再現条件とテスト案は `docs/archive/phases/phase5-followup-fixes.md` に保存済み。
4. 旧Render APIはロールバック先として残し、削除・停止は機能実装と分けて判断する。

- 母・娘・支援者の `recorded_by` と `actor | supporter | target` を、既存画面の操作ごとに固定した。
- target schemaのmigration、master seed、検証、backup、production rollbackの実行ゲートを固定した。
- D2〜D5の保持移行は代替経路として残すが、refresh pathと画面移行開始の必須条件にはしない。

### Final Cleanup: 旧構造の整理

旧テーブル、旧API、旧Render APIは対象ごとに次を測定し、全条件を満たしたものだけ個別に整理する。

- データ保持pathでは新旧集計が一致する。DR-031のrefresh pathではbackup/export記録とseed後件数を確認し、新旧比較を要求しない。
- アプリケーションコードを検索し、migration・テストfixture・archiveを除く旧構造参照が0件である。
- 互換コードに読取・書込カウンタを設け、本番で14日間連続して利用0件である。
- 旧Render APIはhealth checkを除くアクセスが14日間0件である。
- 削除前バックアップ、復元確認、ロールバック手順があり、対象を1つずつ戻せる。

完了条件:

- 上記の測定結果、観測期間、削除対象、復元手順がリリース記録に残る。
- 整理後も対象画面または現行APIを1回動作確認できる。

## 5. 当面の到達目標

当面は「記録できるアプリ」から、次の状態へ移行する。

```text
母と娘の操作
→ 共通の出来事として安全に保存
→ 母と娘それぞれのタイムラインを表示
→ 予定と実際の差を確認
→ 支援・待機・回復の理由を整理
→ 支援者へ必要な範囲だけ説明
→ 次の担当と行動を決める
```

アクセス保護（Phase A）と DB統合の target schema、API-first SPA 同一オリジン配信は完了した。
次は Product Phase 1（予定と実績の差・支援者向け説明）と、必要に応じた Phase F（派生キャッシュ除去）へ進む。

## 6. 実装計画の更新ルール

- 機能を開始するときは、本書の該当Phaseを必要な範囲だけ確認する。WIPメモがある場合は対象ファイルだけ読む。
- 完了した項目は、本書で「完了」または「運用確認待ち」へ更新する。
- 後から参照する必要がある設計方針を変える場合は、`docs/design-decisions.md` へ短いDRを追加し、その後本書を更新する。
- データ構造を変える場合は、`docs/data-model.md` とDB実装計画も同時に更新する。
- コード変更は、対象画面の操作1回または代表的な `curl` 1本が成功したら完了とする。全品質ゲートは要求しない。
- 本番反映が必要な項目は、ローカル完了と本番確認完了を分けて記録する。
- 個別作業の完了後は、同じセッションで作業用文書を `docs/archive/` へ移し、恒久的な結論だけを本書とDRに残す。
