# くらしリレー 実装計画

更新日: 2026-07-19

この文書は、くらしリレー全体の実装状況、未実装領域、開発順序を示す恒久ドキュメントである。
今後「実装計画」「開発計画」「次に何を作るか」を確認するときは、まず本書を参照する。

個別機能の詳細仕様は `docs/wip/<機能>/`、DB移行の詳細手順は
`docs/wip/database-unification/implementation-plan.md`、設計判断は
`docs/design-decisions.md`、データ構造の正は `docs/data-model.md` を参照する。

## 1. 現在地

くらしリレーは、毎日の記録に使えるPoCまで進んでいる。

- 娘の「くらしのおしごと」と母の「家事手帖」は、回数、取消、ポイント・ごほうびをDBへ保存できる。
- 日付別・母娘別の記録を閲覧できる。
- 母用の声かけ画面は、声かけ、完了状態、取消、再通知時刻をDBへ保存できる。
- 娘用ホームは、今日・明日の見通し、夏休みモード、「ママと決めた」、振り返りをDBへ保存できる。
- React SPAとLaravel APIは本番公開済みである。

一方、プロダクトの最終目的である「予定と実績の差を把握し、支援者との役割分担につなげる」流れは未完成である。
ホーム周辺には固定データと画面内だけの操作が残り、予定入力、支援引き継ぎ、支援者向けレポート、
アクセス保護、Google Calendar、通知配信は未実装である。

## 5. 当面の到達目標

当面は「記録できるアプリ」から、次の状態へ移行する。

```text
母と娘の操作
→ 共通の出来事として安全に保存
→ 母と娘それぞれのタイムラインを表示
→ 予定と実際の差を確認
→ 支援・待機・回復の理由を整理
→ 支援者へ必要な範囲だけ説明
→ 次の担当と行動を決める
```

DB統合、履歴保持、アクセス保護を優先し、その安全ゲートを通過した後にレポートと予定機能を追加する。

## 6. 実装計画の更新ルール

- 機能を開始するときは、本書の該当Phaseと `docs/wip/<機能>/` の実装指示書を確認する。
- 完了した項目は、本書で「完了」または「運用確認待ち」へ更新する。
- 設計方針を変える場合は、先に `docs/design-decisions.md` へDRを追加し、その後本書を更新する。
- データ構造を変える場合は、`docs/data-model.md` とDB実装計画も同時に更新する。
- 品質ゲートを実行せず、コードが存在するだけで「完了」としない。
- 本番反映が必要な項目は、ローカル完了と本番確認完了を分けて記録する。
- 個別作業の完了後は、作業用文書を `docs/archive/` へ移し、恒久的な結論だけを本書とDRに残す。
