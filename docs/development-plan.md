# くらしリレー 実装計画

更新日: 2026-07-19

この文書は、くらしリレー全体の実装状況、未実装領域、開発順序を示す恒久ドキュメントである。
今後「実装計画」「開発計画」「次に何を作るか」を確認するときは、まず本書を参照する。

個別機能の詳細仕様は `docs/wip/<機能>/`、DB移行の詳細手順は
`docs/wip/database-unification/implementation-plan.md`、設計判断は
`docs/design-decisions.md`、データ構造の正は `docs/data-model.md` を参照する。

## 1. 現在地

くらしリレーは、毎日の記録に使えるPoCまで進んでいる。

- 娘の「くらしのおしごと」と母の「家事手帖」は、回数、取消、ポイント・ごほうびをDBへ保存できる。
- 日付別・母娘別の記録を閲覧できる。
- 母用の声かけ画面は、声かけ、完了状態、取消、再通知時刻をDBへ保存できる。
- 娘用ホームは、今日・明日の見通し、夏休みモード、「ママと決めた」、振り返りをDBへ保存できる。
- React SPAとLaravel APIは本番公開済みである。

一方、プロダクトの最終目的である「予定と実績の差を把握し、支援者との役割分担につなげる」流れは未完成である。
ホーム周辺には固定データと画面内だけの操作が残り、予定入力、支援引き継ぎ、支援者向けレポート、
アクセス保護、Google Calendar、通知配信は未実装である。

## 4. 実装の優先順位

- **Gate**: 後続Phaseへ進む前に満たす安全・仕様上の関門。
- **Phase**: 基盤を作るデプロイ可能な実装単位。
- **Product Phase**: 基盤の安全ゲート通過後に利用価値を追加する実装群。

以下は原則として記載順に実行する。完了済みのGate・Phaseは、残る運用確認だけを明記する。

### Gate 0: 本番確認を閉じる

状態: **完了（2026-07-19）**。DR-028の機能優先方針と非致命リスク受容に基づく。

完了根拠:

- 本番で主要APIの読取・書込、JST時刻、夏休み計画の母画面反映を確認し、重大な不整合を解消した。
- ローカル品質ゲートと本番スモークが通過した。詳細は
  [Gate 0 本番確認記録](archive/phases/gate0-production-verification-2026-07-19.md) に保管した。

Gate 0後も観測を続けるが、機能実装を止めない項目:

1. Laravel Cloud画面での `migrate:status` 出力保存。既存行の時刻補正から差分migrationの効果は確認済み。
2. 娘さんとの実端末、機内モード、日跨ぎ、朝・昼・夜、10回報酬の観測。
3. H1〜H3/M1〜M3の根治。再現条件とテスト案は `docs/archive/phases/phase5-followup-fixes.md` に保存済み。
4. 旧Render APIはロールバック先として残し、削除・停止は機能実装と分けて判断する。

### Gate 1: 最新DB設計と実装計画を一致させる

DR-027により、`activity_events` を生活上の出来事の共通ヘッダーとし、人物参加、結果、声かけ固有情報、
取消、予定との対応を従属表へ分ける方針へ更新された。

状態: **完了（2026-07-19）**。DR-027との文書同期に加え、DR-031・DR-032で次を確定した。

- 既存少量データは全量export/backupを取得したうえで、管理されたDB refreshを標準経路とする。
- 母・娘・支援者の `recorded_by` と `actor | supporter | target` を、既存画面の操作ごとに固定した。
- target schemaのmigration、master seed、検証、backup、production rollbackの実行ゲートを固定した。
- D2〜D5の保持移行は代替経路として残すが、refresh pathとInertia開始の必須条件にはしない。

正本は `docs/wip/database-unification/implementation-plan.md` の「8.1 Gate 1確定事項」とする。

### Phase B: 低リスク制約と人物FK

状態: **ローカル実装完了・本番確認残り**。拡張マイグレーションはPR #31でmainへマージ済み。

残作業:

- nullableで追加した列の実データ監査
- 本番バックフィル確認
- 契約フェーズでのNOT NULL・最終UNIQUE制約
- PostgreSQL 17での部分インデックス・制約確認

完了条件:

- 本番の既存件数が変わらず、対象列のNULL・重複・不正値が0件である。
- PostgreSQL 17上で期待するUNIQUE・部分インデックス・外部キーの存在を確認できる。
- Seeder再実行後も `routine_templates` が増殖しない。

### Phase A: 家族共有トークン

状態: **Phase Cより前に実装する**。公開中の個人データAPIを無認証のままDB統合期間中も運用しない。
この前倒しはDR-029で、DR-026の旧実行順を改訂する。

安全なデプロイ順:

1. フロントを先に公開し、共通API clientのヘッダ付与、401時の「あいことば」入力、保存解除・更新導線を用意する。
2. Laravel Cloudへ `FAMILY_TOKEN` を設定する。値はGit・フロント環境変数・ビルド成果物へ置かない。
3. バックエンドで `/api/health` 以外を `X-Family-Token` により保護し、連続失敗へRate Limitを適用する。
4. 本番で未設定なら保護APIを起動失敗または503とし、認証を素通ししない。

完了条件:

- トークンなし・不正トークンでは個人データAPIの読取・更新ができず、正しいトークンでは既存操作ができる。
- 401後の再入力、保存解除、トークン更新を実機相当の画面幅で確認できる。
- フロント成果物を検索しても固定トークンが含まれない。
- 対象となる保護経路を、代表的な画面操作または `curl` 1本で確認できる。

### Phase C: 活動マスタ統合

状態: **Phase A完了後の最優先実装**。Inertia移行より先にDB統合を継続する。

1. `activity_definitions` を作成する。
2. `task_definitions` と `routine_templates` をACTキーで接続する。
3. 対応不明な活動を自動統合せず、レビュー一覧にする。
4. 表示名・ラベルの正本を活動マスタへ集約する。

完了条件:

- 全有効タスク・ルーチンが活動マスタへ接続される。
- 同じ活動の名称・ラベルが1か所で管理される。
- 対応不明一覧が明示され、レビューなしの自動統合が0件である。

### Phase D: 共通の出来事・予定・報酬軸

DR-027とDR-031に従い、Inertia開始前は次のtarget schemaを導入する。

- `activity_events`
- `activity_event_participants`
- `activity_event_outcomes`
- `activity_event_cancellations`
- 声かけ固有情報としての `prompt_events`
- `planned_activities`
- 予定と出来事の対応を保持する `plan_actual_links`
- `reward_rules` / `reward_transactions`

既存テーブルは同じリリースで削除しない。現在のデータ量では、バックフィル、二重書込、新旧比較、
旧APIの書込切替をInertia開始条件にせず、必要ならrefresh後にInertiaを新しい書込経路として接続する。

完了条件:

- 母と娘の記録が同じ出来事モデルからそれぞれのタイムラインへ表示できる。
- 声かけ、支援、活動完了を目的別に区別できる。
- 取消後も元の出来事と履歴が残る。
- 空または検証用DBでmigration/seedが通り、必須制約・FK・indexが存在する。
- 予定と出来事を `plan_actual_links` で接続でき、未対応の予定・出来事も失わずに区別できる。

### Phase E: 娘の見通しと予定の接続

- 質問を `plan_questions` で管理する。
- 回答を `plan_answer_versions` へ追記保存する。
- 「明日なにする?」など、活動になる回答だけを `planned_activities` へ接続する。
- 回答変更前後、入力者、誰と決めたかを追跡できるようにする。

完了条件:

- 回答の変更前後、入力者、決め方を履歴から復元できる。
- 活動になる回答だけが `planned_activities` へ接続され、単なる自由記述は予定へ誤変換されない。
- 空または検証用DBで回答履歴と予定接続schemaのmigration/seedが通る。
- 現行API互換adapterと旧データbackfillはInertia開始条件にせず、データ保持を選ぶ場合だけ追加する。

### Gate 2: Inertia移行開始ゲート

状態: **未通過**。次を全て満たすまで、Inertiaのコード実装、パッケージ導入、デプロイ変更を開始しない。

- Phase C〜Eで、統合後DBのテーブル、制約、FK、Seederが定義されている。
- 空または検証用DBでmigration、seed、可能な範囲のrollbackが通る。
- 既存少量データの全量export/backupを取得し、対象環境を明示してDB refreshの直前確認を得ている。
- refreshする場合のバックアップ、実行手順、直前リリースへ戻す手順がある。
- スキーマ検証結果、データ取扱い、残課題、Inertia開始可否をGate記録へ残している。

二重書込、全件バックフィル、新旧データ・集計比較、数日間の本番安定観測は開始条件にしない。
旧テーブルの物理削除も同じリリースで行わず、Inertia移行後に利用状況を見て別判断する。

### Inertia I0〜I7: ハイブリッド構成への移行

状態: **Gate 2通過後に開始**。通常Web画面はLaravel+Inertia+Reactへ移し、オフライン再送、通知、
Service Worker、将来のネイティブ/外部クライアントなど必要性が明確なJSON endpointだけをAPIとして残す。
詳細な開始条件、完了条件、ロールバック条件は `wip/inertia-migration/implementation-plan.md` を正とする。

1. I0: 設計、Inertiaバージョン、API分類、URL・認証・ブランチ・デプロイ方針を確定する。
2. I1: Laravelアプリ内へInertia+React+Vite基盤を追加し、既存Render SPA/APIを残したまま成立させる。
3. I2: 低リスクな読取画面1つを試験移行する。
4. I3: AppShell、共通レイアウト、画面遷移をページ単位で移す。
5. I4: 通常フォームをInertia+Form Requestへ移し、再送・冪等性が必要な操作は内部APIへ残す。
6. I5: 通常Web画面をLaravel session認証へ移し、残存APIの認証方式を用途別に決める。
7. I6: Laravel Cloudの単一配信へ切り替え、Renderをロールバック先として安定観測する。
8. I7: 利用・参照が0件の旧境界コードと旧endpointだけを個別に整理する。

I6の本番安定確認後は新しい業務機能を再開できる。I7の待機期間と安全な整理は並行して継続してよい。

完了条件:

### Product Phase 2: 外部連携と運用強化

1. Google Calendar取込と変更履歴

### Final Cleanup: 旧構造の整理

旧テーブル、旧API、旧Render APIは対象ごとに次を測定し、全条件を満たしたものだけ個別に整理する。

- アプリケーションコードを検索し、migration・テストfixture・archiveを除く旧構造参照が0件である。
- 互換コードに読取・書込カウンタを設け、本番で14日間連続して利用0件である。
- 旧Render APIはhealth checkを除くアクセスが14日間0件である。
- 削除前バックアップ、復元確認、ロールバック手順があり、対象を1つずつ戻せる。

完了条件:

- 上記の測定結果、観測期間、削除対象、復元手順がリリース記録に残る。
- 整理後も対象画面または現行APIを1回動作確認できる。

## 5. 当面の到達目標

当面は「記録できるアプリ」から、次の状態へ移行する。

```text
母と娘の操作
→ 共通の出来事として安全に保存
→ 母と娘それぞれのタイムラインを表示
→ 予定と実際の差を確認
→ 支援・待機・回復の理由を整理
→ 支援者へ必要な範囲だけ説明
→ 次の担当と行動を決める
```

まずアクセス保護を完了し、統合後DBのスキーマ、制約、Seeder、migrationを最優先で完成させる。
本番の安定観測を待たずにInertia中心+必要箇所だけAPIのハイブリッド構成へ移行し、単一配信の確認後に
レポートと予定機能を追加する。

## 6. 実装計画の更新ルール

- 機能を開始するときは、本書の該当Phaseを必要な範囲だけ確認する。WIPメモがある場合は対象ファイルだけ読む。
- 完了した項目は、本書で「完了」または「運用確認待ち」へ更新する。
- 後から参照する必要がある設計方針を変える場合は、`docs/design-decisions.md` へ短いDRを追加し、その後本書を更新する。
- データ構造を変える場合は、`docs/data-model.md` とDB実装計画も同時に更新する。
- コード変更は、対象画面の操作1回または代表的な `curl` 1本が成功したら完了とする。全品質ゲートは要求しない。
- 本番反映が必要な項目は、ローカル完了と本番確認完了を分けて記録する。
- 個別作業の完了後は、同じセッションで作業用文書を `docs/archive/` へ移し、恒久的な結論だけを本書とDRに残す。
