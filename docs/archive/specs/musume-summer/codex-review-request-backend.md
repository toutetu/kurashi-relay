## 重点的に見てほしい点

### A. 設計意図が守られているか(最重要)

DR-021 の核心は「**質問カードを増やしてもカラムが増えない形にする**」ことです。次を確認してください。

1. `daily_plans` に、設問数に比例して増える性質のカラムが**新たに入っていないか**。
   (`start_decided_with` の1本だけは仕様どおりの特例。それ以外にあれば指摘)
2. 決定状態(decided/undecided)が**どこにも永続化されていない**こと。
   導出でなく保存している箇所が残っていたら指摘。
3. `decided_with` が**カテゴリ単位で一貫している**こと
   (`replaceItems` が delete → 再作成する際、全行に同じ値が入る)。

### B. 正しさ

4. `replaceItems()` から旧 state 更新ロジックが漏れなく消えているか。
   消し残しや、逆に**消しすぎて別の副作用を壊していないか**。
5. `titles: []`(全削除)のとき `decided_with` が正しく消えるか。
6. `PATCH` で `wake_up_time` と `start_decided_with` を同時更新したときの挙動。
   片方だけ送られた場合に、もう片方が意図せず消えたり残ったりしないか。
   **特に `wake_up_time` を null クリアしたのに `start_decided_with` が 'mama' のまま残る**
   ような不整合が起きないか。
7. `getSummary()` の `decided_with` が「先頭行の値」を採る実装になっているか。
   行が無いときに `null` を返すか。`tomorrow_plans` が `today_tasks` と同じ作りか。
8. `formatPlanResponse()` の `items` に `tomorrow_plan` キーが**常に存在する**か
   (行が0件でも空配列を返すか。キー自体が消えるとフロントが壊れる)。
9. トランザクション境界・`lockForUpdate` の使い方が既存のまま保たれているか。

### C. テスト

10. `backend/tests/Feature/Api/Musume/MusumePlanTest.php` が
    スペック §5 の1〜5(6は取り下げ)を**実際に検証できているか**。
    アサーションが緩すぎて実質何も見ていないテストがあれば指摘。
11. 上の B6・B8 のような**壊れやすい箇所にテストが無い**なら、それを指摘してください。

### D. 事故チェック

12. `migrate:fresh` 前提の破壊的変更が、**musume以外のテーブル・機能**に波及していないか。
13. seeder / fixture に旧カラム参照の残骸が無いか。
