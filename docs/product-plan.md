# 1. コンセプト

母と娘の生活、予定、実際の活動、娘への支援、
待機・拘束、回復、予定への影響を少ない操作で記録し、
家族・学校・支援機関との役割分担につなげるWebアプリ。

記録自体を目的にせず、次の循環を作る。

```text
生活と予定を記録
→ 予定と実績の差を確認
→ 差が生じた原因を支援・待機・回復として把握
→ 娘の希望と必要な支援を確認
→ 支援者ごとの次の行動を決める
→ 実施結果を確認して再調整
```

# 2. 最上位目標

母が頑張り続けることで成立する生活から、
複数の大人が役割を分担し、
娘も安全な範囲で自分の希望を選べる生活へ移行する。

# 3. 解決したい問題

## 3.1 母の支援時間が見えにくい

登校できた、遅刻した、欠席したという結果だけでは、
次の負担が見えない。

- 起床の声かけ
- 着替えの声かけ
- ゲーム終了の声かけ
- 学校への状況連絡
- 服薬の声かけ
- 持ち物確認
- 腹痛・トイレ対応
- 送迎
- 校内付き添い
- 先生への引き渡し確認
- 夜の聞き取り
- 学校へ伝える内容の整理

## 3.2 就労準備が減った理由が誤解されやすい

「生活リズムが整っていない」
「空いている時間に通所できる」
と単純化せず、予定していた就労準備が何に置き換わったかを示す。

## 3.3 支援の役割が母へ戻りやすい

家族や支援者が担当する予定だった支援について、
担当、条件、完了条件、実施結果、母への差し戻しを残す。

## 3.4 娘の希望が母の判断に埋もれやすい

娘本人の希望、母の観察、母の推測、支援者の聞き取りを分ける。

- PC：詳細修正、分析、レポート

# 7. 予定と実績の基本表示

```text
縦方向：時間の流れ
左側：予定
右側：実績
下または右端：差分と原因
```

予定がない時間も空白にはしない。

- 予定なし＋実績あり：予定外の活動
- 予定なし＋実績なし：実績記録なし

# 8. 技術方針

- 現在: Laravel REST API + Render上の独立React SPA
- DB target schema完成後の目標: Laravel + Inertia.js + Reactを中心とする同一オリジン・単一デプロイ
- 残すAPI: offline再送、冪等記録、Service Worker、通知、native/外部clientに必要なJSON endpoint
- Google Calendar API: Laravelが外部APIを利用する処理としてInertiaと独立して継続
- React
- TypeScript
- Vite
- Tailwind CSS
- モノレポ
- TanStack Query、Zod、API Resourceは一括削除せず、用途がなくなった対象だけ整理する

# 9. デザイン

赤・黄・青・白を同程度に使用し、明るくかわいく、
多くの情報を読み分けやすくする。

## 娘向け
