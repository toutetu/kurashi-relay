# 設計判断の記録(Design Decision Record)

「なぜこの実装にしたか」を後から辿れるようにする記録。
このアプリは生活支援アプリであると同時に就活ポートフォリオでもあるため、
実装の背景にあった**課題感**を残し、将来のアップデート時に立ち返れるようにする。

## 書き方

区切り(機能実装・デザイン方針・技術選定)ごとに、以下の形式で**上に追記**する。

```markdown
## DR-XXX: タイトル(YYYY-MM-DD)

- **課題感**: そのとき何に困っていたか・何を実現したかったか
- **選択肢**: 検討した案(採らなかった案も)
- **決定**: 選んだもの
- **理由**: なぜそれを選んだか(トレードオフ含む)
```

長くなる場合は詳細を別ファイル(スペック等)に書き、ここからリンクする。

## DR-042: 家族共有トークンによるAPI保護を再接続し、Phase Aを完了する(2026-07-20)

- **課題感**: SPA移行中は DR-035 で未認証公開を維持したが、本番APIが公開URLにある以上、
  個人データの読取・書込を保護しない状態を続けるリスクが大きい。実装は PR `#78` で戻り、
  あいことばUIも `#79`〜`#81` で整ったのに、恒久文書と WIP が「未実装／別課題」のまま残っていた。
- **選択肢**: (a) DR-035の未認証公開を継続する / (b) Sanctum等の本格認証へ一気に移る /
  (c) 家族共有トークン（`X-Family-Token`）を `/api/health` 以外へ再接続し、端末内あいことば入力で運用する。
- **決定**: (c)。`EnsureFamilyToken` を `routes/api.php` に再適用。未設定は503（fail closed）、
  不正は401、連続失敗は429。フロントは共通API clientでヘッダ付与し、401時にあいことば必須ダイアログを開く。
  値は半角英数字専用・平文表示。WIPは `docs/archive/specs/family-token/` へ移す。DR-035の
  「保護は別課題」は本DRで置き換える（SPA移行当時の事実記録としては残す）。
- **理由**: 単一家庭ではユーザー個別認証より共有あいことばが運用負荷が低い。SPA同一オリジン配信後も
  `/api/*` は残るため、health以外の保護が必要。UI先行→環境変数→API保護の順で操作不能期間を避けた。

## DR-040: 声かけ書込を activity_events へ接続する(2026-07-20)

- **課題感**: 声かけ完了は `activity_events` に残るが、声かけ POST は `prompt_events` のみで、
  共通出来事・人物参加の正本とずれていた。
- **選択肢**: (a) `prompt_events` を正本のまま維持する / (b) `activity_events` のみに寄せて
  `prompt_events` を廃止する / (c) 声かけ1回ごとに `activity_events`（`event_type=prompt`）と
  参加行を作り、`prompt_events` は固有情報として `activity_event_id` で接続する。
- **決定**: (c)。母=`actor`・娘=`target`、`source=koekake`、冪等キーは
  `koekake-prompt:{idempotency_key}`。予定がある場合は `plan_actual_links` も作る。
  取消は `prompt_events.cancelled_at` と `activity_event_cancellations` を同一TXで追記する。
  `prompt_count` / `latest_prompt_at` キャッシュ更新は当面維持する。
- **理由**: DR-027/036の共通出来事軸と Gate 1 の人物役割規則に揃えつつ、既存API応答と
  一覧キャッシュを壊さない。

## DR-039: おしごと書込を activity_events へ接続し、件数は events+孤児 records で数える(2026-07-20)

- **課題感**: きろく一覧は `activity_events` 正本(DR-036/037)なのに、おしごと POST は `task_records`
  だけを書いていたため、新規タップがタイムラインに出なかった。一方で DR-037 の
  `task_records + activity_events` 合算のまま二重書きすると件数が二重になる。
- **選択肢**: (a) 一覧だけ `task_records` に戻す / (b) おしごとも `activity_events` のみ書き、
  `task_records` を即やめる / (c) `activity_events` を正本として二重書きし、件数は
  `activity_events` + 対応イベントが無い孤児 `task_records` だけを数える。
- **決定**: (c)。POST は同一トランザクションで `task_records`（報酬・取消互換）と
  `activity_events`（`source=oshigoto`、冪等キー `oshigoto:{idempotency_key}`）+ `actor` 参加者を作る。
  DELETE は `task_records.cancelled_at` に加え `activity_event_cancellations` を追記する。
  `last_record_id` は従来どおり `task_records` のみ。DR-037 の暫定合算は本DRで置き換える。
- **理由**: タイムラインと書込を一致させつつ、報酬集計と取消 API を壊さない。孤児加算で
  移行前の `task_records` のみ行と koekake 由来イベントも過不足なく数える。

## DR-037: きろく一覧は activity_events を正本として読む(2026-07-20)

- **課題感**: 声かけ完了は `activity_events` に保存されるが、きろく・おしごと件数は `task_records` だけを
  見ていたため、完了しても画面に出なかった。
- **選択肢**: (a) 完了時に `task_records` へ二重書き込みする / (b) 一覧・件数 API を `activity_events` 基準に切り替える /
  (c) ホーム dashboard まで同時に DB 連動化する。
- **決定**: (b)。`GET /api/task-records` は当該日の `event_type=activity` かつ参加者 `actor` のイベントを返す。
  `GET /api/tasks` の件数と `today_done_count` は `task_records` に加え、同じ条件の `activity_events` を合算する。
  おしごと取消用の `last_record_id` は `task_records` のみとする。dashboard のフィクスチャ連動は別課題。
- **理由**: DR-036の正本と読み経路を一致させ、互換二重書きより移行が明確になる。

> **更新(DR-039)**: おしごと書込が `activity_events` へ接続されたため、件数の暫定合算は
> `activity_events` + 対応イベントが無い孤児 `task_records` に置き換えた。`last_record_id` は本DRのまま。

## DR-036: 活動実績は activity_events の存在で表し、activity_event_outcomes を廃止する(2026-07-20)

- **課題感**: 娘の活動に `completed` / `partial` / `deferred` / `unknown` の結果行を持つと、
  「起きた事実」と「状態ラベル」が混ざり、起床のような瞬間の出来事でも結果テーブルが必要になる。
  先送りや一部実施は実績イベントではなく、再通知・予定変更として扱いたい。
- **選択肢**: (a) `activity_event_outcomes` を維持し完了時に結果行を書く /
  (b) 活動が起きたときだけ `activity_events`（`event_type=activity`）を作り、結果テーブルを廃止する。
- **決定**: (b)。実績の正本は `activity_events` の存在とする。参加者で「一緒にした」「代行」を表し、
  起床など本人しかできない活動は代行を選べない。`ended_at` を nullable で追加し、瞬間の出来事は NULL、
  期間のある活動は `ended_at >= occurred_at` とする。声かけ完了の冪等キーは
  `koekake:daily-task:{daily_task_id}:completion` をサーバー側で安定生成する。
  移行中は `completion_events` / `daily_tasks.status` を同一トランザクションで二重書き込みしてよい。
- **理由**: 「イベントがある＝起きた」が直感的で、未完了・不明の偽実績を作らない。
  DR-027のうち結果従属表の部分を改訂する。詳細は `docs/data-model.md`。

## DR-035: SPA移行中は現行APIアクセス動作を維持し、保護再設計は別課題とする(2026-07-20)

- **課題感**: DR-024/DR-029は家族共有トークンで個人データAPIを保護する方針だったが、
  `7a8391b` 以降の最新mainではAPI routeへfamily-token middlewareが付いていない。
  SPA移行中にstale testや旧middlewareを根拠に保護を独断再有効化すると、認証再設計と画面配信切替が混線する。
- **選択肢**: (a) A3/A4でfamily-tokenやSanctumを同時に戻す / (b) SPA移行を止めて先に保護だけ直す /
  (c) 現行の未認証動作を事実として固定し、保護再設計は専用DR・専用PRの別課題にする。
- **決定**: (c)。SPA移行中はSanctum、session/CSRF追加、`EnsureFamilyToken`のAPI再接続、
  `X-Family-Token`必須化の復活を行わない。公開範囲とリスクは記録するが、このPhaseで解決したと扱わない。
  詳細な現状snapshotは `docs/archive/phases/api-first-spa-migration/access-contract-a3.md` を正とする。
- **理由**: 画面配信方式の切替と認証境界の再設計を同時に変えると、障害原因とロールバック対象を分離できない。
  ユーザーがA4前の保護完了を必須と判断した場合は、本移行を停止して別課題を先に完了する。

## DR-034: Inertia中心方針をやめ、API-first React SPAをLaravel同一オリジンで配信する(2026-07-20)

- **課題感**: DR-030では通常画面をInertia props/formへ移すハイブリッド構成を将来方針としたが、実装が進むと
  Inertiaはrouteとasset配信の器に留まり、実データの取得・更新はTanStack Queryと既存JSON APIのままだった。
  その一方で `frontend/src` と `backend/resources/js` の二重ソース、Inertia専用wrapper、文書上の
  「Gate 2未通過・Inertia未着手」記述が残り、保守負担と方針の食い違いが大きくなった。
- **選択肢**: (a) DR-030どおりInertia props/formへ寄せ続ける / (b) 独立SPA+別デプロイへ戻す /
  (c) React Router + TanStack Query + Laravel JSON APIのまま、SPAをLaravel Cloudから同一オリジン配信し、
  React正本を `backend/resources/js` へ一本化する。
- **決定**: (c)。Inertiaを廃止し、通常画面のデータ取得・更新は既存 `/api/*` に統一する。
  独立React SPAとLaravel APIの2デプロイへ恒久的に戻さず、SPAをLaravelから配信する。
  DR-030は当時の判断として残すが、将来方針は本DRが置き換える。移行手順の正本は
  `docs/archive/phases/api-first-spa-migration/implementation-plan.md` とする。
- **理由**:
  - テスト過重はDR-033とCursor中心実装で解消済みであり、Inertiaへ寄せてJSON契約テストを減らす主因は薄れた。
  - 現Inertiaはprops/formをほぼ使わず、APIを全件維持しているため、Inertia固有の利点が小さい。
  - React source二重化の方が保守負担になった。
  - client主導のoffline再送・冪等処理(`idempotency_key`、localStorage退避)を維持するには、
    API-firstのまま同一オリジン化する方が自然である。

## DR-033: 個人利用の機能追加は動作確認1回で完了し、作業文書と古いDRをアーカイブする(2026-07-19)

- **課題感**: 機能が増えるたびに、自動テストの追加、全テスト・lint・build、別AIレビュー、修正後の再レビュー、
  詳細な実装指示書とレビュー依頼書が積み重なり、個人利用アプリの機能追加より検証と文書作成に時間と
  AIトークンを使う状態になっていた。さらに毎回複数の恒久資料、完了済みWIP、全DRを読む運用も文脈を肥大させていた。
- **選択肢**: (a) 従来の品質ゲートを全変更で維持 / (b) 自動テストだけ減らし、独立レビューと全文読込は維持 /
  (c) 通常変更は動作確認1回で完了し、復旧コストが高い変更だけレビュー1周、文書と読込も必要最小限にする。
- **決定**: (c)。新機能に自動テストを書かず、テスト通過を完了条件にしない。画面を1度操作するか代表的な
  `curl` を1本実行して対象機能が動けば完了とする。既存テストの修正は任意・後回しでよい。
  レビューは原則行わず、DBスキーマ・データ移行、認証・権限、データ削除・一括更新、本番デプロイ設定などだけ
  実装者以外が1周確認する。原則1機能=1セッションとし、数行で足りる指示は文書化しない。
  WIPは複数セッションにまたがる未完作業だけに使い、完了時にarchiveへ移す。通常は関連箇所だけを部分的に読み、
  古いDR本文は20件程度のまとまりで `docs/archive/design-decisions/` へ移して索引から必要なものだけ参照する。
- **理由**: 利用者が本人だけのPoCでは、通常機能の回帰リスクより、機能が完成せず日常で使えない損失と
  AI利用コストの方が大きい。確認をゼロにはせず、実際の利用経路を1回通すことで「動く」を最短で確かめる。
  一方、データ消失・情報漏えい・復旧不能は個人利用でも損失が大きいため、高リスク変更だけ独立確認を残す。
  この決定はDR-017のテスト方針、DR-014の文書運用、DR-001の単一DRファイル運用をこの範囲で更新する。

## DR-032: Gate 1は管理下refreshと画面起点の人物役割・復旧ゲートを正本化して完了する(2026-07-19)

- **課題感**: DR-027の従属表分離は文書へ同期済みだったが、少量の既存データを保持移行するか、画面操作から
  `recorded_by` と `actor | supporter | target` をどう決めるか、productionでrefreshに失敗した場合に何へ戻すかが
  未確定だった。このままではGate 1が設計未確定に見え、Cursorへ一意な実装指示を渡せない。
- **選択肢**: (a) D2〜D5を必須として既存データを全件移行 / (b) backupなしでDBを作り直す /
  (c) 全量backup後の管理下refreshを標準とし、役割は画面操作から固定、production recoveryはbackup restoreとする。
- **決定**: (c)。既存少量データはrefresh直前に全量export/backupし、新DBへ自動importしない。seedは人物、活動、
  ルーチン、質問、報酬規則等の安定したマスタだけに限定する。入力者は `recorded_by_member_id`、生活上の関与は
  参加行に分け、娘のおしごと=娘actor、母の家事=母actor、声かけ=母actor+娘target、一緒に実施=娘actor+母supporter、
  母が代行=母actor+娘targetとする。`deferred` / `unknown` は娘の予定活動に対する未完了の状態として、
  母recorded_by+娘actorで記録する。migration/seedの空DB検証、全量backup、対象環境とrestore手順の記録、
  実行直前の明示確認をrefresh開始条件とする。失敗時は破壊的migrationのdownではなく直前コード+backup restoreへ戻す。
- **理由**: 現在のデータ量では完全な互換移行より、復元可能性を確保して正しい箱を早く作る方が目的に合う。
  一方、refreshを無断削除にせず、全量backupと実行直前確認を分離することで、文書上の方針決定と実際の破壊的操作を
  混同しない。役割を画面から自動決定すれば1タップ操作を保ち、入力者と実行者も混同しない。詳細な操作表と
  実行ゲートは `docs/archive/phases/database-unification/implementation-plan.md` の8.1を正とする。
  Gate 2 クローズ後の保管先は同ディレクトリ（DR-041）。

## DR-030: DB統合完了後にInertia中心のハイブリッド構成へ段階移行する(2026-07-19)

- **課題感**: SPA+REST分離を維持したまま通常のWeb画面を追加すると、API client、Zod schema、
  TanStack Query hook、Form Request、API Resource、API Controller、CORS、Base URL、通信エラー、
  キャッシュ無効化、2デプロイ間の契約互換を毎回扱う必要がある。現時点でAPIは18 endpoint、
  API境界に関係するコードは概算2,339行あり、実運用で機能を追加・回収する速度に合わなくなった。
- **選択肢**: (a) REST分離を維持してテストだけを減らす / (b) DB統合を止めてInertiaへ一括移行 /
  (c) 統合後DBのスキーマを先に確定し、その後、通常画面はInertia、必要な処理だけAPIとして残す
  ハイブリッド構成へ段階移行する。
- **決定**: (c)。DB統合は中断せず最優先で継続し、Inertia実装と同時進行しない。ただしInertia開始前に
  必要なのは統合後DBのテーブル、制約、FK、Seeder、migrationの成立までとする。現在の本番データは少量のため、
  二重書込、新旧データ・集計比較、長期の本番安定観測は開始条件にしない。DBの箱を作り直したら直ちにInertia I0へ進む。
  通常の画面遷移・初期データ・フォーム送信はLaravel+Inertia+Reactへ移す。一方、オフライン退避・再送、

## DR-027: activity_eventsを生活の出来事の中心にし、人物参加・結果・取消を従属表へ分離する(2026-07-19)

- **課題感**: 母のクイック記録、娘のおしごと、声かけ、支援、家事が別々の記録体系にあると、
  最終的に見たい「私のタイムライン」と「娘のタイムライン」を同じ基準で組み立てられない。
  また `activity_events` に `actor` / `supporter` / `target` を個別のnullable FKとして持たせる案は、
  1人だけの活動で不要列がNULLになり、同じ場面に複数人が関わる場合の拡張もしにくい。
- **選択肢**: (a) 既存どおり声かけ・完了・おしごとを別テーブルのまま集計 / (b) `activity_events` に
  人物役割を固定nullable列で持つ / (c) `activity_events` を出来事ヘッダーとし、人物は
  `activity_event_participants`、結果・声かけ固有情報・予定対応・取消は必要なときだけ従属行で持つ。
- **決定**: (c)。1タップで実際に起きたことを `activity_events` に1件記録し、関わった人物と役割を
  `activity_event_participants` の `actor | supporter | target` で1件以上記録する。
  声かけも `event_type=prompt` の出来事とし、回数・文面等だけを1対1の `prompt_events` に置く。
  活動結果は `activity_event_outcomes`、取消は `activity_event_cancellations`、予定との対応は
  `plan_actual_links` に分離し、存在しない概念をNULL列で表さない。入力者は出来事ヘッダーの
  `recorded_by_member_id` に必須で記録する。母画面と娘画面は分離したまま、保存先を共通化する。
- **理由**: 1つの共同活動を複製せず、娘=`actor`・母=`supporter` の同じ出来事として両方の
  タイムラインへ表示できる。別時刻の支援と娘の実施は別イベントとして正確に残せる。
  UIは人物や役割を選ばせず、画面とボタンから自動設定するため1タップ原則を維持できる。
  出来事ヘッダーを必須情報だけに絞ることで、役割・結果・予定・取消の「該当なし」をNULLで表す必要がない。
  この判断はDR-019 / DR-023のうち「完了と声かけを共通イベントへ接続しない」
  「`activity_events` は完了結果のみ」「`prompt_events` は独立した正本」という部分を改訂する。
  活動マスタ・共通予定・報酬台帳・追記保存・段階移行の方針は維持する。詳細は
  `docs/data-model.md`。

## DR-022: 運用開始後のスキーマ変更は差分ALTERで行い、CREATE書き換え方針(DR-013)を終了する(2026-07-18)

- **課題感**: 夏休み対応(PR #27/#28)を本番マージした後、リリース手順書を書く過程で本番の
  `php artisan migrate:status` を確認したところ、`daily_plans` / `plan_items` /
  `reflection_sessions` が **バッチ[2]で "Ran" 済み**だった。DR-013 の方針で**ファイル名を変えずに
  CREATEを書き換えた**ため、Laravel は「適用済み」と判断して二度と実行しない。つまり本番は
  **旧スキーマのまま**(`*_state` 3本が残り、`decided_with` / `start_decided_with` が無い)。
  DR-021 に「DBを破棄・再作成した」と書いたが、**実行されたのはローカルだけで本番では未実施**だった。
  さらに厄介なことに、この状態は**エラーとして表面化しなかった**。APIリソースは存在しないカラムを
  `null` で返すため、curl では `start_decided_with: null` / `decided_with: null` と、
  「まだ誰も使っていない正常な状態」と見分けがつかない。夏休み機能は本番で無言のまま機能していなかった
  (「🎀 ママと決めた」の書き込みだけが落ちる)。加えてこの間に娘が実データを入力しており
  (2026-07-18分の見通し)、DR-013 が前提にした「実データが無い」条件はすでに崩れていた。
- **選択肢**:
  - A: 手順書どおり `migrate:fresh --force` を打つ。→ 今日の実データが消える。かつ DR-013 が自ら定めた
    「**実運用開始をもってこの方針は終了する**」という終了条件に反する。
  - B: **差分ALTERマイグレーションを1本追加**し、通常の `migrate --force` で当てる。
  - C: 本番にだけ手でSQLを流す。→ 本番のスキーマがマイグレーション履歴から導出できなくなり、
    環境の再現性を失う。
- **決定**: **A(ただし「最後の一度」として)**。`migrate:fresh --force` + `db:seed --force` を実行し、
  **DR-013 の CREATE書き換え方針はここで終了**する。以後、適用済み環境があるテーブルの変更は
  **必ず差分ALTERで積む**(ファイル名を変えずにCREATEを書き換えることを禁止)。
  手順書: `docs/ops/musume-summer-release.md`。
- **理由**:
  - **失うデータを実測したら、ほぼ無かった**(2026-07-18 実測)。
    - おしごと: `child` / `mother` とも `lifetime_count` 0・コレクション0・コイン/ポイント0。
      **ゾンビ図鑑の蓄積はゼロ**(DR-012 で娘のモチベーションの核としたもの。ここに蓄積があれば
      判断は逆だった)。
    - 声かけ: タスク22件はすべてシード由来のテンプレート生成物で、`prompt_events` は**0件**。
    - 実データは**今日の見通し4件のみ**(入浴 / 英語の宿題 / 夏休みの宿題 / 水筒)。ユーザー承認済み。
  - 当初は B(ALTER)を推した。根拠は**本番データの保全**だったが、保全すべき対象がほぼ無いと
    実測で判明した時点でこの根拠は消えた。冪等ALTERを1本書いてCodexレビューに回すコストと、
    **夏休み機能が今日中に動き出す価値**を比べ、後者を取った。夏休みの真っ最中で、
    娘が今まさに使う機能である点が効いている。
  - B の本来の価値は「本番データを保全したままスキーマを変えられる体制」の獲得だが、
    **今回それを検証する対象データがそもそも無い**。体制は次の変更で本番のデータを相手に作るほうが実質的。
  - C はスキーマの由来がマイグレーション履歴から追えなくなり、ポートフォリオとしても
    「本番だけ手当てされた不明な状態」を抱える。
  - fresh 後は CREATE ファイルと本番DBが再び一致し、今回の乖離は解消する。**乖離が再発しない条件は
    「CREATEをもう書き換えないこと」**であり、それは fresh か ALTER かと独立に、この DR で確定させた。
- **旧 `*_state` の値は移行しない**: fresh なので自動的に消えるが、仮に ALTER を選んでいた場合も
  バックフィルは**してはいけなかった**。旧 `with_mama` は「娘が決めかねたので**ママと決める**
  (先送り・**未**決定)」で、新 `decided_with='mama'` は「**ママと決めた**(決定済み)」。DR-021 の
  運用変更で**意味が反転している**ため、機械的に移すと「まだ決めていない」を「決めた」と記録してしまう。
  同じ列名・同じ値でも意味が変わっていることがある、という点は次回のALTER時に効くので残しておく。
- **副次的な学び**: 「CREATEを書き換える」は**全環境を fresh できる間だけ**成立する運用で、
  適用済み環境が1つでも残った瞬間に、マイグレーション履歴とファイル内容が乖離する。しかも
  **乖離は例外を投げない**——`Schema::create` はスキップされ、APIは欠けた列を `null` で返し、
  挿入も旧列の default が埋めるので通ってしまう。検知に効いたのは `migrate:status` の
  **バッチ番号**だけだった(fresh なら全部バッチ[1]に揃う。分かれている=freshは走っていない)。
  API応答の目視は当てにならない、を教訓として残す。

## DR-021: 決定状態カラムを廃止し、「ママと決めた」を状態でなく属性として持つ(2026-07-18)

- **課題感**: 3つあった。
  1. 運用開始前日に運用方針が変わった。娘が決めかねたとき「ママと決める(先送り)」ではなく、
     **ママが聞き取ってその場で一緒に決め、代理入力する**運用になった。ボタンの意味が
     「ママと決める」→「**ママと決めた**」に変わり、未決定ではなく決定済みを指すようになった。
  2. 夏休みモードに「明日なにする?」カードを足したかった。
  3. そのとき、`daily_plans` が**質問カード1枚につき1カラム**(`today_state` /
     `tomorrow_items_state` / `start_state`)を持つ形だと気付いた。ユーザーから
     「ママと決める項目ごとに1カラム使っていたら、今後このページの項目を追加・変更できない気がする」
     と指摘があり、そのとおりだった。カードを増やすたびにスキーマが変わる。
- **選択肢**:
  - A: `tomorrow_plan_state` カラムを追加する。→ カードを増やすたびにマイグレーションが要り、
    指摘された問題がそのまま残る。
  - B: `answer_states` の json カラム1本にまとめる。→ 破綻はしないが、中身がほぼ空の器を
    維持することになる。
  - C: **`*_state` カラムを廃止して決定状態は値から導出し、「決め方」だけを属性として持つ。**
- **決定**: C。`daily_plans` から `*_state` 3本を削除。`plan_items` に `decided_with`
  (nullable / `'mama'` or null)を1本、`daily_plans` に `start_decided_with` を1本追加。
  差し引きでカラムは1本減る。スペックは `docs/archive/specs/musume-summer/musume-summer-spec.md`。
- **理由**:
  - 実装を確認したら、`*_state` は `MusumePlanService::replaceItems()` が
    `$titles === [] ? 'undecided' : 'decided'` と**項目の有無をそのまま書き写している**だけで、
    固有情報を持つのは `with_mama` のときだけだった。その `with_mama` は運用変更で消える。
    つまり3本とも**すでに導出値**であり、削っても情報を失わない。
  - 「今は決めない」は元から何も保存していないので、`undecided` と未タッチの区別も存在しなかった。
    廃止による情報損失はゼロ。
  - 「ママと決めた」は `undecided` / `decided` と**並ぶ状態ではなく**、決定済みに付随する
    **属性(誰と決めたか)**。属性を `plan_items` の列に置けば、カードが増えても増えるのは
    **行**であって列ではない。ユーザーの懸念(項目を追加・変更できなくなる)が構造的に解消する。
  - 「明日 何時に起きる?」だけは値が `plan_items` ではなく `wake_up_time` というスカラ列に入るため
    `start_decided_with` を特例で1本持つ。設問数に比例して増える列ではないので許容した。
    将来この設問も `plan_items` に寄せれば統一できるが、移行コストに見合わないので今回はしない。
  - まだ実運用を開始していないため、DR-013 の方針どおり**既存のCREATEマイグレーションを編集**し、
    DBを破棄・再作成した。マイグレーションを増やさずに済んだ。
    > **訂正(2026-07-18)**: 破棄・再作成を実行したのは**ローカルのみ**で、本番では実施されていなかった。
    > 本番は旧スキーマのまま残り、夏休み機能が無言で機能しない状態になった。経緯と対処は **DR-022** を参照。
- **副次的な学び**: 「カラムを1本足すだけ」で通せる変更でも、**足す先の形が設問数に比例して
  伸びるなら設計の問題**。ユーザーの「今後、項目を追加・変更できない気がする」という直感が
  正しく、実装を読み直したら削れる根拠(すでに導出値だった)まで出てきた。
