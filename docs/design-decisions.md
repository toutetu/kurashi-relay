# 設計判断の記録(Design Decision Record)

「なぜこの実装にしたか」を後から辿れるようにする記録。
このアプリは生活支援アプリであると同時に就活ポートフォリオでもあるため、
実装の背景にあった**課題感**を残し、将来のアップデート時に立ち返れるようにする。

## 書き方

区切り(機能実装・デザイン方針・技術選定)ごとに、以下の形式で**上に追記**する。

```markdown
## DR-XXX: タイトル(YYYY-MM-DD)

- **課題感**: そのとき何に困っていたか・何を実現したかったか
- **選択肢**: 検討した案(採らなかった案も)
- **決定**: 選んだもの
- **理由**: なぜそれを選んだか(トレードオフ含む)
```

長くなる場合は詳細を別ファイル(スペック等)に書き、ここからリンクする。

## DR-003: 完了記録から操作履歴とポイント単価を分離する(2026-07-17)

- **課題感**: 同じ完了へ別の冪等キーで到達した操作が保存されず、取消後の再送で完了が復活する経路があった。
  また、現在のタスク単価から過去ポイントを再計算すると、単価変更時に過去の獲得値まで変わってしまう。
- **選択肢**: (a) 完了記録だけで冪等性とポイントを管理し続ける / (b) 受理操作と完了記録を分け、
  完了時単価を記録へ保存する / (c) ポイント残高だけを別カラムで保持する
- **決定**: (b) 全受理キーを `task_record_operations` に保存してcanonicalな完了記録へ結び、
  `task_records.granted_point_value` に完了時単価を保存する。
- **理由**: 操作の再送安全性と業務上の完了状態を分離でき、取消後も受理済み操作を同じ記録へ解決できる。
  ポイントは履歴から導出する方針を維持しながら、当時の単価を再現できる。残高カラム方式より不整合の原因が少ない。
  詳細: `docs/deploy_change/oshigoto-persistence-fix-spec-02.md`

## DR-002: アプリ名の表記は「くらしリレー」に決定(2026-07-16)

- **課題感**: アプリ名を「くらしリレー」と「くらしリレイ」のどちらの表記にするか迷いがあった。
- **選択肢**: (a) くらしリレー(標準の長音表記) / (b) くらしリレイ(イ止めでブランド調。
  「レイ」=光/Ray の連想があり、きらきら要素や紫ゴスロリの世界観とは相性が良い)
- **決定**: (a) くらしリレー
- **理由**: アプリのコンセプトが「家事のバトンを家族でつなぐ」ことそのものなので、
  意味が瞬時に伝わる分かりやすさを最優先した。長音「ー」は音・見た目ともにバトンが
  続いていく感じと合い、子どもから祖父母まで一目で読める。検索・ストアでも「リレー」で
  打たれる。(b)は誤記に見える恐れと「霊」連想の小リスクがあり、その洗練・光の気配は
  ロゴやキャッチコピーで補う方針。既存のリポジトリ名・docs表記とも一致し手直し不要。

## DR-001: 設計判断を記録する運用を始める(2026-07-16)

- **課題感**: 実装を進めるうちに「なぜこうしたか」が失われていく。就活ポートフォリオとして
  見せるとき・将来アップデートするときに、当初の課題感を思い出す手掛かりがない。
- **選択肢**: (a) コミットメッセージのみに頼る / (b) 1判断1ファイルの厳密なADR / (c) 単一ファイルに軽量に追記
- **決定**: (c) この `design-decisions.md` に軽量フォーマットで追記していく
- **理由**: コミットメッセージは「何をしたか」で「なぜか」が残りにくい。厳密なADRは個人開発には
  重すぎて続かない。1ファイルなら一覧性が高く、ポートフォリオとしてそのまま見せられる。
