# 設計判断の記録(Design Decision Record)

「なぜこの実装にしたか」を後から辿れるようにする記録。
このアプリは生活支援アプリであると同時に就活ポートフォリオでもあるため、
実装の背景にあった**課題感**を残し、将来のアップデート時に立ち返れるようにする。

## 書き方

区切り(機能実装・デザイン方針・技術選定)ごとに、以下の形式で**上に追記**する。

```markdown
## DR-XXX: タイトル(YYYY-MM-DD)

- **課題感**: そのとき何に困っていたか・何を実現したかったか
- **選択肢**: 検討した案(採らなかった案も)
- **決定**: 選んだもの
- **理由**: なぜそれを選んだか(トレードオフ含む)
```

長くなる場合は詳細を別ファイル(スペック等)に書き、ここからリンクする。

## DR-027: activity_eventsを生活の出来事の中心にし、人物参加・結果・取消を従属表へ分離する(2026-07-19)

- **課題感**: 母のクイック記録、娘のおしごと、声かけ、支援、家事が別々の記録体系にあると、
  最終的に見たい「私のタイムライン」と「娘のタイムライン」を同じ基準で組み立てられない。
  また `activity_events` に `actor` / `supporter` / `target` を個別のnullable FKとして持たせる案は、
  1人だけの活動で不要列がNULLになり、同じ場面に複数人が関わる場合の拡張もしにくい。
- **選択肢**: (a) 既存どおり声かけ・完了・おしごとを別テーブルのまま集計 / (b) `activity_events` に
  人物役割を固定nullable列で持つ / (c) `activity_events` を出来事ヘッダーとし、人物は
  `activity_event_participants`、結果・声かけ固有情報・予定対応・取消は必要なときだけ従属行で持つ。
- **決定**: (c)。1タップで実際に起きたことを `activity_events` に1件記録し、関わった人物と役割を
  `activity_event_participants` の `actor | supporter | target` で1件以上記録する。
  声かけも `event_type=prompt` の出来事とし、回数・文面等だけを1対1の `prompt_events` に置く。
  活動結果は `activity_event_outcomes`、取消は `activity_event_cancellations`、予定との対応は
  `plan_actual_links` に分離し、存在しない概念をNULL列で表さない。入力者は出来事ヘッダーの
  `recorded_by_member_id` に必須で記録する。母画面と娘画面は分離したまま、保存先を共通化する。
- **理由**: 1つの共同活動を複製せず、娘=`actor`・母=`supporter` の同じ出来事として両方の
  タイムラインへ表示できる。別時刻の支援と娘の実施は別イベントとして正確に残せる。
  UIは人物や役割を選ばせず、画面とボタンから自動設定するため1タップ原則を維持できる。
  出来事ヘッダーを必須情報だけに絞ることで、役割・結果・予定・取消の「該当なし」をNULLで表す必要がない。
  この判断はDR-019 / DR-023のうち「完了と声かけを共通イベントへ接続しない」
  「`activity_events` は完了結果のみ」「`prompt_events` は独立した正本」という部分を改訂する。
  活動マスタ・共通予定・報酬台帳・追記保存・段階移行の方針は維持する。詳細は
  `docs/data-model.md`。

## DR-026: 4日間はDB統合の中核を最優先し、追加機能は安全ゲート後に着手する(2026-07-18)

- **課題感**: 木曜日までの4日間で、夏休み機能の確認、3系統に分かれたDBの統合、支援者向け情報提供などの
  未実装機能をできる限り進めたい。一方、DB統合の全Phaseと複数画面を同時に完成させようとすると、
  本番データ移行・履歴・互換APIの検証が薄くなり、同じ活動の二重管理を残したまま機能だけが増える。
- **選択肢**: (a) 画面追加を優先 / (b) API保護を最初に完了 / (c) 夏休み機能の残確認を短時間で閉じた後、
  DB統合の中核を優先し、安全に完了した範囲に応じて追加機能へ進む。
- **決定**: (c)。実行順は、夏休み機能の本番確認、低リスク制約と安定キー、活動マスタ、共通予定・実績軸、
  娘の見通し接続の順とする。家族共有トークンはDB中核の直後かつ支援者向けレポートより前に実装する。
  支援者向けレポート、予定入力、支援引き継ぎ、設定は品質ゲートを通過した残時間で順に着手する。
- **理由**: 最大の懸念は同じ活動と完了が別体系に存在することであり、先に共通軸を作ると、その後の予定・レポート・
  Google Calendarが同じ正本を利用できる。夏休み対応の本番 `migrate:fresh` は実施済みなので再実行せず、
  残る書き込み確認だけを先頭の短いゲートとして扱う。詳細は
  `docs/wip/database-unification/execution-brief.md`。

## DR-025: routine_templatesに名称から独立したslugを追加する(2026-07-18)

- **課題感**: `KoekakeSeeder` は `routine_templates` を (`phase`, `name`) で `updateOrCreate` している。
  表示名を変更すると同じルーチンの更新ではなく新しい行が作られ、既存 `daily_tasks` と別系統になる。
  `activity_key` は朝食と夕食、朝と夜の歯磨きなどで意図的に共通なので、ルーチン行の一意キーにはできない。
- **選択肢**: (a) (`phase`, `name`) をUNIQUE / (b) `activity_key` をUNIQUE / (c) 安定した `slug` を追加。
- **決定**: (c)。差分ALTERで `slug` を追加し、既存22件をバックフィルしてUNIQUE・NOT NULL化する。
  Seederの検索キーも `slug` へ変更し、`name` は変更可能な表示値、`activity_key` は共通活動への対応キーとする。
- **理由**: 表示文言を変えても同じルーチンを更新でき、同じ活動を複数の時間帯・枠で使う設計も維持できる。
  既存CREATEマイグレーションは変更せず、重複監査後に段階的なALTERで導入する。

## DR-024: 単一家庭専用を維持し、家族共有トークンで全個人データAPIを保護する(2026-07-18)

- **課題感**: くらしリレーは我が家専用であり、複数家庭対応の `households` / `household_id` を全テーブルへ
  入れる必要はない。一方、本番APIは公開URLにあり、家庭数が1つでも認証なしなら第三者が記録を閲覧・追加・取消できる。
  CORSはブラウザの読取制御であり、curlや別サーバーからの直接アクセスを防がない。
- **選択肢**: (a) households/users/Sanctumを導入 / (b) 家族共有トークンで保護 / (c) 認証なしを継続。
- **決定**: 複数家庭対応は行わず、人物軸は既存 `family_members` を使う。`/api/health` を除く個人データAPIを
  `X-Family-Token` で保護する。トークンはLaravel環境変数だけに置き、フロントへ固定埋込みせず、利用者が端末で
  入力する。失敗には401とRate Limitを適用し、環境変数の差し替えでローテーション可能にする。
- **理由**: 単一家庭で必要なのはテナント分離ではなく、公開URLから家庭データを守る最小のアクセス制御である。
  本格アカウントより実装・運用が軽く、将来端末単位の失効が必要になった場合だけSanctum等へ移行できる。
  実装順は `docs/wip/database-unification/implementation-plan.md` Phase A。

## DR-023: 3系統を活動マスタ・予定・実績の共通軸で接続し、変更履歴を追記保存する(2026-07-18)

- **課題感**: おしごとの `task_definitions` / `task_records`、声かけの `routine_templates` /
  `completion_events`、娘の `daily_plans` / `plan_items` が独立し、同じ活動と完了が別体系に重複している。
  このままGoogle Calendar予定を追加すると、第4の予定体系が増え、予定と実績の突き合わせが困難になる。
  また完了状態のupsert、plan_itemsの削除再作成、振り返りの1行更新では変更前の履歴が失われる。
- **選択肢**: (a) 現行3系統を維持してAPI集計だけで接続 / (b) 巨大な単一イベントテーブルへ統合 /
  (c) 活動の意味・予定・実績を共通軸にし、声かけ・報酬・回答は目的別テーブルで接続。
- **決定**: (c)。`activity_definitions` を活動の正本、`planned_activities` をルーチン・娘の見通し・手入力・
  Google Calendar共通の予定、`activity_events` をおしごとと完了結果共通の実績正本とする。
  `prompt_events` は支援行為として分離を維持し予定へ接続、ポイントは `reward_transactions` 台帳へ分離する。
  娘の回答は `plan_answer_versions`、Calendarは `calendar_event_versions` で版を追記し、訂正・取消で過去行を消さない。
- **理由**: 「何の活動か」を一度だけ定義し、予定元が違っても同じ予定・実績比較を使える。
  声かけと完了を混ぜない既存原則を守りながら、ポイントや支援量を同じ活動へ関連付けられる。
  一括置換はせず、追加・バックフィル・新旧比較・読取切替・旧書込停止の段階移行とする。
  詳細は `docs/data-model.md` と `docs/wip/database-unification/implementation-plan.md`。

## DR-022: 運用開始後のスキーマ変更は差分ALTERで行い、CREATE書き換え方針(DR-013)を終了する(2026-07-18)

- **課題感**: 夏休み対応(PR #27/#28)を本番マージした後、リリース手順書を書く過程で本番の
  `php artisan migrate:status` を確認したところ、`daily_plans` / `plan_items` /
  `reflection_sessions` が **バッチ[2]で "Ran" 済み**だった。DR-013 の方針で**ファイル名を変えずに
  CREATEを書き換えた**ため、Laravel は「適用済み」と判断して二度と実行しない。つまり本番は
  **旧スキーマのまま**(`*_state` 3本が残り、`decided_with` / `start_decided_with` が無い)。
  DR-021 に「DBを破棄・再作成した」と書いたが、**実行されたのはローカルだけで本番では未実施**だった。
  さらに厄介なことに、この状態は**エラーとして表面化しなかった**。APIリソースは存在しないカラムを
  `null` で返すため、curl では `start_decided_with: null` / `decided_with: null` と、
  「まだ誰も使っていない正常な状態」と見分けがつかない。夏休み機能は本番で無言のまま機能していなかった
  (「🎀 ママと決めた」の書き込みだけが落ちる)。加えてこの間に娘が実データを入力しており
  (2026-07-18分の見通し)、DR-013 が前提にした「実データが無い」条件はすでに崩れていた。
- **選択肢**:
  - A: 手順書どおり `migrate:fresh --force` を打つ。→ 今日の実データが消える。かつ DR-013 が自ら定めた
    「**実運用開始をもってこの方針は終了する**」という終了条件に反する。
  - B: **差分ALTERマイグレーションを1本追加**し、通常の `migrate --force` で当てる。
  - C: 本番にだけ手でSQLを流す。→ 本番のスキーマがマイグレーション履歴から導出できなくなり、
    環境の再現性を失う。
- **決定**: **A(ただし「最後の一度」として)**。`migrate:fresh --force` + `db:seed --force` を実行し、
  **DR-013 の CREATE書き換え方針はここで終了**する。以後、適用済み環境があるテーブルの変更は
  **必ず差分ALTERで積む**(ファイル名を変えずにCREATEを書き換えることを禁止)。
  手順書: `docs/ops/musume-summer-release.md`。
- **理由**:
  - **失うデータを実測したら、ほぼ無かった**(2026-07-18 実測)。
    - おしごと: `child` / `mother` とも `lifetime_count` 0・コレクション0・コイン/ポイント0。
      **ゾンビ図鑑の蓄積はゼロ**(DR-012 で娘のモチベーションの核としたもの。ここに蓄積があれば
      判断は逆だった)。
    - 声かけ: タスク22件はすべてシード由来のテンプレート生成物で、`prompt_events` は**0件**。
    - 実データは**今日の見通し4件のみ**(入浴 / 英語の宿題 / 夏休みの宿題 / 水筒)。ユーザー承認済み。
  - 当初は B(ALTER)を推した。根拠は**本番データの保全**だったが、保全すべき対象がほぼ無いと
    実測で判明した時点でこの根拠は消えた。冪等ALTERを1本書いてCodexレビューに回すコストと、
    **夏休み機能が今日中に動き出す価値**を比べ、後者を取った。夏休みの真っ最中で、
    娘が今まさに使う機能である点が効いている。
  - B の本来の価値は「本番データを保全したままスキーマを変えられる体制」の獲得だが、
    **今回それを検証する対象データがそもそも無い**。体制は次の変更で本番のデータを相手に作るほうが実質的。
  - C はスキーマの由来がマイグレーション履歴から追えなくなり、ポートフォリオとしても
    「本番だけ手当てされた不明な状態」を抱える。
  - fresh 後は CREATE ファイルと本番DBが再び一致し、今回の乖離は解消する。**乖離が再発しない条件は
    「CREATEをもう書き換えないこと」**であり、それは fresh か ALTER かと独立に、この DR で確定させた。
- **旧 `*_state` の値は移行しない**: fresh なので自動的に消えるが、仮に ALTER を選んでいた場合も
  バックフィルは**してはいけなかった**。旧 `with_mama` は「娘が決めかねたので**ママと決める**
  (先送り・**未**決定)」で、新 `decided_with='mama'` は「**ママと決めた**(決定済み)」。DR-021 の
  運用変更で**意味が反転している**ため、機械的に移すと「まだ決めていない」を「決めた」と記録してしまう。
  同じ列名・同じ値でも意味が変わっていることがある、という点は次回のALTER時に効くので残しておく。
- **副次的な学び**: 「CREATEを書き換える」は**全環境を fresh できる間だけ**成立する運用で、
  適用済み環境が1つでも残った瞬間に、マイグレーション履歴とファイル内容が乖離する。しかも
  **乖離は例外を投げない**——`Schema::create` はスキップされ、APIは欠けた列を `null` で返し、
  挿入も旧列の default が埋めるので通ってしまう。検知に効いたのは `migrate:status` の
  **バッチ番号**だけだった(fresh なら全部バッチ[1]に揃う。分かれている=freshは走っていない)。
  API応答の目視は当てにならない、を教訓として残す。

## DR-021: 決定状態カラムを廃止し、「ママと決めた」を状態でなく属性として持つ(2026-07-18)

- **課題感**: 3つあった。
  1. 運用開始前日に運用方針が変わった。娘が決めかねたとき「ママと決める(先送り)」ではなく、
     **ママが聞き取ってその場で一緒に決め、代理入力する**運用になった。ボタンの意味が
     「ママと決める」→「**ママと決めた**」に変わり、未決定ではなく決定済みを指すようになった。
  2. 夏休みモードに「明日なにする?」カードを足したかった。
  3. そのとき、`daily_plans` が**質問カード1枚につき1カラム**(`today_state` /
     `tomorrow_items_state` / `start_state`)を持つ形だと気付いた。ユーザーから
     「ママと決める項目ごとに1カラム使っていたら、今後このページの項目を追加・変更できない気がする」
     と指摘があり、そのとおりだった。カードを増やすたびにスキーマが変わる。
- **選択肢**:
  - A: `tomorrow_plan_state` カラムを追加する。→ カードを増やすたびにマイグレーションが要り、
    指摘された問題がそのまま残る。
  - B: `answer_states` の json カラム1本にまとめる。→ 破綻はしないが、中身がほぼ空の器を
    維持することになる。
  - C: **`*_state` カラムを廃止して決定状態は値から導出し、「決め方」だけを属性として持つ。**
- **決定**: C。`daily_plans` から `*_state` 3本を削除。`plan_items` に `decided_with`
  (nullable / `'mama'` or null)を1本、`daily_plans` に `start_decided_with` を1本追加。
  差し引きでカラムは1本減る。スペックは `docs/wip/musume-summer/musume-summer-spec.md`。
- **理由**:
  - 実装を確認したら、`*_state` は `MusumePlanService::replaceItems()` が
    `$titles === [] ? 'undecided' : 'decided'` と**項目の有無をそのまま書き写している**だけで、
    固有情報を持つのは `with_mama` のときだけだった。その `with_mama` は運用変更で消える。
    つまり3本とも**すでに導出値**であり、削っても情報を失わない。
  - 「今は決めない」は元から何も保存していないので、`undecided` と未タッチの区別も存在しなかった。
    廃止による情報損失はゼロ。
  - 「ママと決めた」は `undecided` / `decided` と**並ぶ状態ではなく**、決定済みに付随する
    **属性(誰と決めたか)**。属性を `plan_items` の列に置けば、カードが増えても増えるのは
    **行**であって列ではない。ユーザーの懸念(項目を追加・変更できなくなる)が構造的に解消する。
  - 「明日 何時に起きる?」だけは値が `plan_items` ではなく `wake_up_time` というスカラ列に入るため
    `start_decided_with` を特例で1本持つ。設問数に比例して増える列ではないので許容した。
    将来この設問も `plan_items` に寄せれば統一できるが、移行コストに見合わないので今回はしない。
  - まだ実運用を開始していないため、DR-013 の方針どおり**既存のCREATEマイグレーションを編集**し、
    DBを破棄・再作成した。マイグレーションを増やさずに済んだ。
    > **訂正(2026-07-18)**: 破棄・再作成を実行したのは**ローカルのみ**で、本番では実施されていなかった。
    > 本番は旧スキーマのまま残り、夏休み機能が無言で機能しない状態になった。経緯と対処は **DR-022** を参照。
- **副次的な学び**: 「カラムを1本足すだけ」で通せる変更でも、**足す先の形が設問数に比例して
  伸びるなら設計の問題**。ユーザーの「今後、項目を追加・変更できない気がする」という直感が
  正しく、実装を読み直したら削れる根拠(すでに導出値だった)まで出てきた。

## DR-020: K2娘用ホームは「見通し3項目」を主役にし、できたこと記録・ポイント連動はK3へ送る(2026-07-18)

- **課題感**: K2の画面コンセプトが2案あった。計画のD-01は「見通し3項目」(いまから何する?/明日何がいる?/
  明日何時間目から?)中心、ChatGPT docx設計書のF-01は「できたこと記録」中心で、画面の主役が異なる
  (DR-019 §9-3で「K2設計時に必ず確認」とした宿題)。また docx の時間帯「いつでも」の扱いも未決だった。
- **選択肢**: (a) 見通し3項目中心 / (b) 折衷(見通し主役+できたこと簡易記録) / (c) できたこと記録中心。
  いつでも枠は (x) K2で導入 / (y) K3へ見送り。
- **決定**(ユーザー確認 2026-07-18): **(a) 見通し3項目中心**。できたこと記録とポイント連動

## DR-017: API分離構成(SPA+REST)を維持し、Inertiaへは戻さない(2026-07-17)

- **課題感**: 前作(Inertiaモノリス+Cursor単独実装)に比べ、テスト・検証に大幅に時間がかかっている。
  原因が (1) フロント/バックをREST APIで分離した構成のせいか、(2) スペック→実装(Cursor)→
  レビュー(Codex)→修正指示→再レビューという多段プロセスのせいかを切り分けたかった。
  機能はまだ欲しいものの1/5程度で今後も開発が続くため、「作り直すなら小さい今のうち」という
  タイミング論も含めて、Inertiaへ戻すべきかを判断する必要があった。
- **選択肢**: (a) 今Inertiaへ移行する(モノリス化・セッション認証・1デプロイ化) /
  (b) API分離を維持し、検証コストはテスト自動化とプロセス改善で削る /
  (c) 判断を保留し、再評価の条件(ゲート)だけ決めておく
- **決定**: (b)+(c)。API分離を維持する。検証はプロセス側で軽量化する:
  ① API新機能はPestのfeatureテストを実装の完了条件に含め、docsベースのcurlスモークは
  デプロイ時確認(コールドスタート・本番設定)に限定する
  ② フロントはvitest+MSWで契約を検証し、実機確認はリリース前の1回に絞る
  ③ 小差分(目安200行以下)はレビュー1周で切り上げる
  ④ API+Webを跨ぐ機能でも実装指示書は1枚に統合する(ブランチ/PRは2本のまま)。
  **再評価ゲート**: 実運用開始後の次の大区切り(K3レポート完了、または本格認証が必要になった時点)で、
  残機能がフォーム/帳票CRUD中心なら Inertia 移行を再検討する。
- **理由**:
  - **遅さの主因は構成ではなくプロセスだった**。task-count 1機能でレビュー往復3周・関連docs 8枚・PR 2本。
    多段委譲はトークン費用を下げるが、リードタイムは往復回数に比例して伸びる。ここを直さない限り
    Inertiaに戻しても速くならない。
  - **このアプリの核はクライアント主導の同期設計**。「連打・電波不安定下でのタップ記録」(DR-015/016)が
    本質で、楽観的更新+冪等キー+補償キャンセル+localStorage退避という設計は本番検証済みの最難関部分。
    サーバ主導のInertiaモデルではここを作り直し・再検証することになる。Reactコンポーネント自体は
    Inertiaでも流用できるため「全部書き直し」ではないが、生き残るのは簡単な部分で、死ぬのは一番
    苦労した部分という逆転が起きる。
  - **完成済みの1/5はまさにこの同期基盤**であり、残り4/5はその上に乗る。1機能あたりの限界コストは
    今後下がる(task-countは冪等処理・ごほうび計算をそのまま再利用できた)。「残りが4倍あるから
    構成の税金も4倍」という単純計算にはならない。
  - **サーバはシンガポール**(DR-006)。Inertiaは画面遷移ごとにサーバ往復(約0.1秒+)を踏むが、
    SPAは遷移がローカルで完結し、通信はデータ取得時だけ。スマホでのサクサク感に直結する。
  - **今後の声かけリマインダー等の通知系**は、ブラウザの外からサーバが独立して動く必要があり、
    APIサーバがある構成の方が素直に拡張できる。
  - **移行は数週間の機能フリーズを伴い**、「実運用に足りる機能追加を最優先」(DR-016)と正面から矛盾する。
  - 補足: 現状の接続は「APIキー方式」ではなく**認証なし**(DR-007の意図的な判断)。Inertia化は
    セッション認証(ログイン画面)の新規実装を強制するため、スコープ削減ではなく追加になる。
    認証が必要になったら、まず保存済みの `X-Family-Token` 方式(DR-007)を検討する。

## DR-015: 完了記録をトグルからカウント(1タップ=+1)へ変える(2026-07-17)

- **課題感**: 現行のタスク記録は1日1回のトグルで、2回目のタップが取り消しになる。しかし記録したい
  行動(声かけに応える・お手伝い等)は1日に何回もするものが中心で、トグルでは記録が実態と合わず、
  善意のタップで記録が消えてしまう。
- **選択肢**: (a) `task_records` に回数カラムを持たせて increment / (b) 1タップ = 1行のイベントログにする
  (「1日1タスク1件」の部分ユニークインデックスを外す)/ (c) トグルを維持し「×3回やる」等の
  回数別タスク定義を増やす。
- **決定**: (b)。押すたびに `task_records` が1行増え、その日のカウント = 有効レコード件数。
  取り消しは専用の小さな「−」ボタンで最後の1件をキャンセル。
- **理由**: タップごとの時刻が残り、将来の声かけレポート(K3)の素材になる。冪等キーによる二重送信防止と
  ごほうび計算(有効レコード件数ベース)がそのまま使える。(a)は履歴が消え冪等処理が複雑化、(c)は定義の
  無限増殖を招く。副作用としてゲージが「タスク10個」→「タップ10回」で進むが、何回もやる動作をそのまま
  褒める形になるため意図どおりとする(速すぎれば `stamp_size` の config 調整で対応)。
  スペック: `docs/archive/specs/task-count/task-count-spec.md`。
- **追記(2026-07-17 実装完了時)**: UIは実機確認したユーザーの指定により、当初案(チェック丸の回数
  バッジ+専用−ボタン)から**ホームの「クイック記録」カードと同じ操作感**(行タップ=+1・「N件」ピル・
  丸「+」・下部トースト「取り消す」5秒)へ変更した(`task-count-ui-quickcards.md`)。

  積み、根治は member 単位の直列キュー化で行う。BE=PR #9 / FE=PR #11 で本番反映済み。

## DR-011: Phase 8を「安定確認(8a)」と「旧Render停止(8b)」に分け、8bはゲート条件までしない(2026-07-17)

- **課題感**: Phase 7で本番フロントをLaravel Cloud APIへ切替済み。Phase 8は「安定運用の確認→旧Render API停止」だが、
  (1) 旧Render APIは現行の唯一のロールバック先(§8)で、止めると切り戻しが遅くなる。
  (2) 並行系の既知不具合(H1〜3/M1〜3・DR-010)が未修正で、実家庭利用の「子どもの連打・モバイル電波不安定」で
  顕在化しうる。安定確認と停止を一気に進めると、問題が出たときに戻せない。
- **選択肢**: (a) 指示書どおり順にこなし、確認できたら旧Renderを止める / (b) 安定確認(8a)と旧Render停止(8b)を
  分け、8bは「8a合格 + 重大修正の本番反映 or リスク受容 + 実家庭で数日無事」を満たすまで着手しない /
  (c) 旧Renderをすぐ止めてコスト/管理を単純化する
- **決定**: (b)。ランブック `docs/deploy_change/phase8-stability-and-render-stop.md` に 8a チェックリスト
  (curl=Codex委譲 / 実機=ユーザー / 観測=日跨ぎ)と 8b の停止手順(Suspend優先・Deleteはユーザー再確認後)を分離。
  8a は Phase 6 の積み残し(休止復帰後の初回POST・10回達成報酬)も回収する。
- **理由**: 安定確認中は必ず戻せる状態を保ちたい。旧Renderは残置コストが小さく(無料/自動休止)、
  ロールバック保険として置く価値が上回る。停止・削除は不可逆寄りの外向き操作なので、ユーザー明示確認をゲートにした。
  委譲方針(検証はcurlスモークをCodex・実機はユーザー、Fableはブラウザ駆動しない)とも一致。全文はランブック参照。

## DR-009: 日報の下書きを毎日20時に自動生成する(2026-07-17)

- **課題感**: 日報を書く作業(フォーマット約160行を読む+コミット履歴を読み手向けに翻訳)がFableのトークンを
  多く消費していた。定期的に発生する定型作業なので、繰り返し部分のコストを下げたい。一方で、作業時間・体調・
  感想は本人にしか分からず、フォーマットの禁止事項も「メモにないことを創作しない」と定めている。
- **選択肢**:
  (a) 毎回Fableが対話で全部書く / (b) 20時に完全自動で日報を完成させる /
  (c) 20時に「下書き」まで自動生成し、人にしか分からない項目は【要確認】で残す
- **決定**: (c)。Windowsタスクスケジューラ `KurashiRelay-DailyLogDraft` が毎日20時に
  `.claude/skills/daily-log/scripts/generate-draft.ps1` を実行。決定的な部分(前回日報の特定・
  対象作業日の判定・git材料の収集・メモ書き出し)はスクリプト(無料)、本文の清書はCursor(最安)に委譲し、
  作業時間・体調・感想・次回予定は【要確認】で残す。仕上げ(数値記入・レビュー・コミット)は人+日報スキル。
- **理由**: (b)は禁止事項と衝突し、作業時間・体調が毎回【要確認】の不完全な日報になる。(a)は毎回コストが高い。
  (c)なら繰り返しコストをほぼゼロ(スクリプト+Cursor)に落としつつ、創作しないルールを守れる。
  委譲体制(実装・量産はCursor/Codex、Fableは設計・レビュー)とも一致する。
  ※スケジュール登録はマシンローカル(タスクスケジューラ)でリポジトリには入らない点に注意。

## DR-008: フロント永続化はサーバsummaryを唯一の真実にする(2026-07-17)

- **課題感**: 現行フロントはゲージを `INITIAL_JAR(6) + 今日の完了数` で数え、満タンで `-10` して折り返し、
  コイン/ポイントもローカル `useState` で加算、報酬ゾンビ/お菓子も `pickRandom*` でローカル抽選していた。
  一方バックエンドは `gauge_count = lifetime_count % 10`(累積・毎日リセットなし)、報酬は節目POST時に
  サーバが `item_slug` を決めて返す設計。両者のゲージの数え方・報酬の決定主体が食い違い、素朴に繋ぐと
  二重加算やゲージ不整合、報酬の二重演出が起きる。
- **選択肢**: (a) フロントの数え方を残し、APIは保存だけに使う(表示はローカル計算のまま) /
  (b) サーバの `summary`(gauge_count/full_count/coins/points)と POSTレスポンスの `revealed_reward` を
  唯一の真実とし、フロントは楽観的更新→レスポンスで確定同期する / (c) 折衷(ゲージはローカル、報酬はサーバ)
- **決定**: (b)。ゲージ/コイン/ポイントは `summary` を表示し**クライアントで加減算しない**。報酬は
  サーバの `item_slug` をローカルの `ZOMBIES`/`SWEETS` から引いて表示。二重表示は `milestone_number` で防止。
  `localStorage` は本番DBの代替ではなく**オフライン退避**とし、冪等キー固定で再送二重登録を防ぐ。
- **理由**: 履歴から導出する残高方針(DR-003/004)と一貫させ、二重加算・折り返し不整合・報酬二重演出を
  構造的に排除できる。冪等・取消後非復活は本番試験(`phase6-results.md` #3,#4,#8)で実証済みで、
  フロントは冪等キーを固定するだけで安全。初期の見せかけ値(`INITIAL_JAR`/`INITIAL_COINS`/`INITIAL_POINTS`)は
  永続化されておらず、接続後は実DB(0始まり)を正とする。持ち越しは `reward_adjustments` へユーザー確認後に
  別途投入(Phase 7相当)。詳細スペック: `docs/deploy_change/oshigoto-persistence-frontend-spec.md`

## DR-006: APIとDBはシンガポールリージョンに揃える(2026-07-17)

## DR-005: APIとDBだけLaravel Cloudへ移し、ReactはRenderに残す(2026-07-16)

  さらに記録がuseStateのみで、更新すると消える(永続化なし)。明日から家族で実利用したい。
- **選択肢**: (a) Render有料化 / (b) Laravel Cloud(API+Serverless Postgres) / (c) AWS(App Runner+RDS)
- **決定**: (b)。ReactはRender Static Siteに残し、フロント切替は環境変数`VITE_API_BASE_URL`のみで

## DR-004: 冪等キーごとの報酬応答と移行時単価を固定する(2026-07-17)

- **課題感**: 節目レコードを別キーで重複受理した場合、初回は報酬なしでも再送時にcanonicalレコードの
  報酬が返り、同じキーの応答が変わっていた。また、単価スナップショット列の追加前に存在する記録は
  デフォルト値0のままとなり、移行を境に過去ポイントが失われた。
- **選択肢**: (a) operationごとにレスポンス全体を保存する / (b) レコード作成キーだけ報酬を返し、
  重複受理キーは常に報酬なしとする / (c) 重複受理でもcanonicalレコードの報酬を常に返す。
  既存単価は0のままにする案と、移行時点のtask definition単価で埋める案も比較した。
- **決定**: (b)を採用し、再送時はoperationのキーとレコード作成キーが一致する場合だけ報酬を返す。
  `granted_point_value` 追加時には、既存記録を紐づくtask definitionの現単価で一度だけバックフィルする。
- **理由**: レスポンス保存用の列を増やさず、同一キーの初回と再送を一致させ、重複操作で古いお祝い演出が
  再表示されることを防げる。既存記録の当時単価は保存されていないため完全復元はできないが、現単価での
  バックフィルが移行直前の集計値を維持できる最も安全な近似である。

## DR-003: 完了記録から操作履歴とポイント単価を分離する(2026-07-17)

- **課題感**: 同じ完了へ別の冪等キーで到達した操作が保存されず、取消後の再送で完了が復活する経路があった。
  また、現在のタスク単価から過去ポイントを再計算すると、単価変更時に過去の獲得値まで変わってしまう。
- **選択肢**: (a) 完了記録だけで冪等性とポイントを管理し続ける / (b) 受理操作と完了記録を分け、
  完了時単価を記録へ保存する / (c) ポイント残高だけを別カラムで保持する
- **決定**: (b) 全受理キーを `task_record_operations` に保存してcanonicalな完了記録へ結び、
  `task_records.granted_point_value` に完了時単価を保存する。
- **理由**: 操作の再送安全性と業務上の完了状態を分離でき、取消後も受理済み操作を同じ記録へ解決できる。
  ポイントは履歴から導出する方針を維持しながら、当時の単価を再現できる。残高カラム方式より不整合の原因が少ない。
  詳細: `docs/deploy_change/oshigoto-persistence-fix-spec-02.md`

## DR-002: アプリ名の表記は「くらしリレー」に決定(2026-07-16)

- **課題感**: アプリ名を「くらしリレー」と「くらしリレイ」のどちらの表記にするか迷いがあった。
- **選択肢**: (a) くらしリレー(標準の長音表記) / (b) くらしリレイ(イ止めでブランド調。
  「レイ」=光/Ray の連想があり、きらきら要素や紫ゴスロリの世界観とは相性が良い)
- **決定**: (a) くらしリレー
- **理由**: アプリのコンセプトが「家事のバトンを家族でつなぐ」ことそのものなので、
  意味が瞬時に伝わる分かりやすさを最優先した。長音「ー」は音・見た目ともにバトンが
  続いていく感じと合い、子どもから祖父母まで一目で読める。検索・ストアでも「リレー」で
  打たれる。(b)は誤記に見える恐れと「霊」連想の小リスクがあり、その洗練・光の気配は
  ロゴやキャッチコピーで補う方針。既存のリポジトリ名・docs表記とも一致し手直し不要。

## DR-001: 設計判断を記録する運用を始める(2026-07-16)

- **課題感**: 実装を進めるうちに「なぜこうしたか」が失われていく。就活ポートフォリオとして
  見せるとき・将来アップデートするときに、当初の課題感を思い出す手掛かりがない。
- **選択肢**: (a) コミットメッセージのみに頼る / (b) 1判断1ファイルの厳密なADR / (c) 単一ファイルに軽量に追記
