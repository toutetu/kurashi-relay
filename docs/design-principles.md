# くらしリレーの設計原則

1. くらしリレーは、母や娘を監視・評価するアプリではない。
2. 母へ集中した支援、待機、拘束、回復を可視化し、役割分担につなげる。
3. 娘本人の発言、母の確認、母の観察、母の推測を区別する。
4. 達成・失敗や頑張り不足という評価を行わない。
5. 日常で最も頻繁に使う操作を画面上部へ置く。
6. ホームでは詳細情報より、すばやい記録と今日の概要を優先する。
7. PCは一覧性と情報密度を優先する。
8. タブレットは日常の主入力端末として扱う。
9. スマートフォンは外出中の即時記録を優先する。
10. 小さい画面ではタブによる段階的開示を使用する。
11. 同じ機能を端末別に複製せず、同じコンポーネントを再配置する。
12. ホームは概要を表示し、詳細は専用画面へ分ける。
13. 主要操作は1〜2タップ、操作領域は44px以上とする。
14. 色だけで状態を表現しない。
15. 操作には押下、処理中、成功、失敗、未保存のフィードバックを返す。
16. 共通コンポーネントとデザイントークンを優先する。
17. 直接カラーコードや画面固有の重複スタイルを増やさない。
18. 200％表示でも操作不能や横スクロールを発生させない。
19. アクセシビリティと可読性を装飾より優先する。
20. 実データや子どものセンシティブ情報を不用意に共有しない。
21. ボタンは用途で選ぶ（主操作 `primary`／副操作 `secondary`／状態選択 `selection`／低優先度 `low`）。見た目名で増やさない。
22. 行リストの追加操作は「＋」だけでなく行全体をボタンにし、動詞付きラベルにする。

## ホームと画面分割

- ホームには頻繁に使用する操作と最小限の今日情報だけを置く。
- 娘の情報は、母のホームへ混在させず専用画面へ分ける。
- 娘本人の入力と母の確認・観察を区別する。
- 予定への影響、対応事項、ゲーム進捗などの詳細情報はまとめ画面へ置く。
- PCホームは `home-grid`(grid-template-areas)で管理し、ズーム100%・高さ約900pxで縦スクロールなしに収める。
- 「現在の活動」は上部の横長NowBarとして常時表示する(カードにしない)。
- PCホームの左1列は「👀 きょうのようす」(今日の予定=1日分すべて)、右2列は「✏️ きろくする」(クイック活動記録・クイック記録)とする。
- 同じ行のカードは高さをそろえ、行をまたいだ固定高さは使用しない。カード自身は横幅を決定しない。
- モバイルでは利用場面に応じて、ホームのカードを「今日」「記録」へ分類する。
- スマホ・タブレットのホームは「今日」「記録」の2タブとする(既定は今日)。
- ラストウォーは生活支援や予定への影響とは性質が異なるため、独立ページで扱う。
- 外部向けレポートや支援者向け資料へラストウォーの詳細を自動的に含めない。
- ゲーム進捗は本人の生活の一部として扱うが、支援評価や就労評価と混同しない。
- 色は確定値ではなく、共通トークンから後で変更できる構造にする。
- 画面固有の直接カラーコードを増やさない。
- 同じ段のカードは高さをそろえ、一覧性を高める。
- 固定高で内容を切らず、段の中で自然に高さを合わせる。
- カードの影は真下に落ちる薄いアンビエント影のみ(斜め方向の影は古い印象になるため使わない)。
- 可読性と情報の区切りは、濃い影ではなく余白、枠線、背景、アクセントで表現する。
- クリック可能な要素は、hover、active、focusの各状態を一貫して表現する(詳細は下の「押し心地」)。
- タッチ端末ではhoverに依存しない。
- 色の状態変化もデザイントークンで管理する。

## B案v3 ビジュアルシステム(2026-07リニューアル)

以降の新画面はすべてこのシステムに従う。正解の見た目は `docs/mockups/home-b3.html`、
実装済みの参照例は ホーム画面(`frontend/src/pages/HomePage.tsx` ほか)。

### 設計判断の背景(なぜこうしたか)

旧デザインの問題: ①カード全面のパステル塗りで幼稚な印象 ②赤黄青紫が同時に主張して色がケンカ
③上辺の太い色ボーダーが古い ④全文字太字で強弱がない ⑤斜め影が素人っぽい。

対策の核: **色の面積を減らすと内容が主役になる**。かわいさは色数ではなく、
丸み・押し心地・小さな演出(ピコッ、+1フライ)で出す。

### 1. きぶんカラー(テーマシステム)

- ユーザーがその日の気分で選ぶ5テーマ: さくら🌸(既定)・あい💙・そら💧・もり🌿・よる🌙
- `html[data-mood="…"]` で `--primary / --primary-deep / --primary-soft / --ground / --line / --line-soft / --ink / --muted / --faint` が切替わる(`frontend/src/styles/tokens.css`)
- **選択状態・主ボタン・リンク・バッジは必ず `--primary` 系を使う**(=きぶんに追従する)
- 新画面で「この画面専用の色」を発明しない

### 2. カテゴリ色(きぶんに依存しない固定色)

意味が固定された情報にだけ使う。使用は「アイコンチップ・小さなドット・バッジ」まで。

| 色 | トークン | 割り当て |
|---|---|---|
| 珊瑚 | `--coral` | ラストウォー、現在の活動(旧・赤系の情報) |
| 山吹 | `--amber` | 家事、クイック記録 |
| 藍系青 | `--cat-blue` | 就労準備、待機、時間・データ系 |
| 藤 | `--fuji` | 娘に関するすべての情報 |
| 緑 | `--green` | 進行中ステータス、登校支援 |

### 3. カードの作法

- 背景は白(`--surface`)、枠は1pxヘアライン(`--line`)、角丸 `--card-radius`、影は薄い真下アンビエント
- カード見出し: 丸いアイコンチップ(30px, カテゴリ色のsoft地+濃色アイコン)+ 13.5px/800
- 太字は「数字」と「見出し」だけ。本文・説明は通常ウェイト+`--muted`
- 数字は `tabular-nums` で桁をそろえる

### 4. 押し心地(全画面共通の必須要件)

押せるものは見ただけで押せるとわかり、押すと気持ちいいこと。

- **立体の縁**: 押せる要素は下2pxの縁(`box-shadow: 0 2px 0 <縁色>`)。キーボードのキーの比喩
- **hover**: `translateY(-1〜-2px)` で浮き、縁が深くなり、薄い影が広がる
- **active**: `translateY(+1.5px)`・縁消失で「沈む」
- **クリック後**: ピコッ(`kr-pop`: scale .93→1.05→1、バウンス系イージング)
- **カウント系**: 値バッジの再マウントで `kr-bump`(ぽよん)+「+1」が飛んで消える(`kr-flyup`)
- `prefers-reduced-motion: reduce` で transform/animation を必ず無効化
- 実装は `frontend/src/index.css` の共通クラス(`.popping` `.count-bump` `.fly` ほか)を再利用する

### 5. 入力ファーストのレイアウト文法

- 画面の目的が「記録」なら、入力UIを最上部・最大面積に置く
- いま進行中のものは細いバー(NowBar型)で常時見せる。大きなカードで場所を取らない
- ゾーン見出し(`✏️ きろくする` / `👀 きょうのようす` 形式)で「操作する場所」と「見る場所」を分ける
- 行リストは**行全体をボタン**にする(＋アイコンは目印であり、当たり判定は行全体)
- スマホは下部ナビ中央の丸FABが最頻操作(きろく)
- 1〜5段階の入力は数字ボタンではなく、ハート(体調)・顔絵文字(気分)で直感的に

### 6. 新画面への応用レシピ

**ラストウォー タスクリスト(母のタスク)**
- アイデンティティ色: 珊瑚(`--coral`)のアイコンチップ。カードは白のまま
- タスク行は「行全体ボタン」型(クイック記録と同じ作法)。完了トグルでピコッ+チェック演出
- 進捗バーは `--primary` 系(きぶんに追従)、プレイ時間・回復評価の数値は tabular-nums の太字
- 「達成/失敗」の評価語は使わない(プロダクト原則)。進捗は事実の可視化として見せる

**娘の気分画面**
- アイデンティティ色: 藤(`--fuji`)。娘のデータであることを色で一貫させる
- 体調=ハート・気分=顔絵文字の入力をそのまま流用(`ScoreControl` の hearts/faces)
- 入力者区分(娘本人/母が確認/母の観察)の区別を必ず見えるところに置く
- 月・星・リボンは小さな装飾アイコン(lucide の Moon / Sparkles 等)として控えめに。面で塗らない
- きぶんカラーはこの画面にも適用される(娘が「よる🌙」を選べば画面全体が藤紫トーンになり、藤アクセントと調和する)

## くらしのおしごと／ママの家事手帖 デザイン方針(2026-07)

「くらしのおしごと」(娘: `/oshigoto`)と「ママの家事手帖」(`/mama-kaji`)に適用する。
正解の見た目は `docs/mockups/child-oshigoto-halloween.html` と `docs/mockups/mama-kaji-sweets.html`。
実装スペックは `docs/oshigoto-frontend-spec*.md`。

### 設計判断の背景(なぜこうしたか)

- **本人の「好き」を核にする。** ごほうび設計は仮説ではなく本人に確認して決める。
  娘の好き＝USJ・ハロウィン・ゾンビ収集・年パスで週1・おこづかいでハロウィングッズ(2026-07本人確認)。
  お菓子テーマは「食べるのは好きだが集めたくはない」で不採用→ママ版に転用した。
  テーマ(スキン)は本人の興味が変わったら差し替えられる構造にしておく。
- **タップの瞬間が主役。** 記録アプリの満足感は「押した瞬間」に集約される。
  完了タップごとに全画面のごほうび演出(びん/月の大写し+ひとこと+「+1」)を出す。
  大きな音は鳴らさない。約2秒で自動で閉じ、連続記録を妨げない。
  10個目は演出の着地を見せてから(約0.8秒)スタンプ獲得へ連結する(演出をモーダルで隠さない)。
- **ごほうびの終着点は本物の体験。** 画面内の達成を現実の家族時間・本人の目標に着地させる。
  娘=おこづかい→USJグッズ(目標は本人が選ぶ)。ママ=ポイント→「おやつを作る券」(親子の台所時間)。

### 世界観スキンの独立

- 娘=紫ゴスロリ×ハロウィン(`--osh-*`)、ママ=パティスリー(`--mkj-*`)。
  各featureのローカルCSS変数で完結させ、`features/oshigoto` と `features/mamakaji` は相互importしない。
- ダッシュボードのカード部品(DashboardCard等)は使わない。世界観が濁るため。
  共通で使うのは `Button`・`AppShell`・フォーカスリング等のアクセシビリティ基盤のみ。
- ライト/ダーク両対応。ダークは「よる」の世界観として作る(単なる色反転にしない)。

### 進捗メタファー

- 数字だけでなく「満ちていく絵」で進捗を見せる。
  娘=本物の月の満ち欠け(斜め下の三日月→半月→十三夜→満月。lit部分を35°傾ける。
  作図式は `docs/oshigoto-frontend-spec.md` のMoonGauge仕様)。
  ママ=おかしびん(完了ごとにふたが開いてキャンディが1粒ころんと落ちる)。
- 繰り越し(10個超過分)を可視化し、「10でリセットされて損」と感じさせない。

### せめない設計(モック要件由来・実装ルール)

- 赤・バツ・減点表示を出さない。未完了は静かな薄色で「まだできるよ」。
- 取り消しはいつでもワンタップ。ただし**取り消し・訂正系のUIは小さく目立たなく**
  (12px薄色テキスト程度。タップ領域だけ44px確保)。主役はお祝いであって保険ではない。
- 声かけされてやっても自発でも同じ1回。評価語(達成/失敗/頑張り不足)を使わない。
- 毎日リセットしない。日をまたいで積み上がる。
- 夜のふりかえりの質問は1つだけ。反省を必ずは聞かない。

### 文言

- 漢字は大人と同じ。表現(言い回し)は小学生らしく。ふりがな目的の全ひらがな化はしない。
- ひとこと褒めはタスクごとに固有にする(「床、きれいになった！」「名もなき家事も、ちゃんと1個！」)。
  汎用の「よくできました」を使い回さない。

### 権利・出典

- USJ等の固有キャラクターの姿・ロゴは使わない。ゾンビ等は「種類・名称」のみをオリジナル絵文字表現で扱う。
- グッズ価格・ライド名等の実データは出典年を明記し、毎年差し替えられるデータ構造にする。
