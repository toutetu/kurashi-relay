# AI実装・レビュー運用(高リスク変更のみ)

更新日: 2026-07-19

## 目的

DBスキーマ・データ移行、認証・権限、データ削除・一括更新、本番デプロイ設定など、
壊すと復旧コストが高い変更だけを実装者以外が1周確認する。

通常の機能追加にはこの手順を使わない。通常は `AGENTS.md` の最小主義に従い、
1機能を1セッションで実装し、画面操作1回または `curl` 1本で完了する。

## 役割

| 役割 | 担当 |
|---|---|
| 実装 | そのセッションで依頼を受けたAI |
| 独立レビュー | 高リスク変更の実装者とは別のAI |

モデルは固定しない。ユーザー指定を優先し、必要以上に高価なモデルへ引き継がない。

## 標準フロー

1. **対象を限定する**
   - 既存の未コミット差分は取り消さない。
   - 今回の高リスク箇所、復旧方法、レビュー対象外を数行で共有する。
   - 数行で足りる場合は実装指示書・レビュー依頼書を作らない。
2. **実装する**
   - 依頼を受けたAIが実装する。別AIへの実装委譲は必須にしない。
   - 変更箇所の動作確認を1回行う。全テスト・lint・typecheck・buildは自動で追加しない。
   - 高リスク箇所に固有の安全確認(バックアップ、対象環境、rollback等)だけ必要最小限で行う。
3. **独立レビューを1周行う**
   - 実装者以外のAIへ、差分と復旧方法を渡す。長いレビュー依頼書は不要。
   - データ消失、認証漏れ、復旧不能など、重大な問題だけを確認する。
4. **完了する**
   - 重大な指摘があれば該当箇所だけ直す。ユーザーが明示しない限り再レビューしない。
   - 変更要点、動作確認1回の結果、高リスク箇所の確認結果、残リスクを短く報告する。

## レビューで見る項目(必要なものだけ)

- 変更対象と対象外が混ざっていないか
- データを失う経路と復旧方法
- 認証・権限・秘密情報の漏れ
- 本番反映時に戻せなくなる変更

## 運用上の注意

- 同じworktreeを複数AIに同時編集させない。レビューは実装完了後に開始する。
- Windows版Cursor Agentはsandboxモード非対応のため、allowlistとAuto-reviewを使う。作業範囲はworktreeと読み取り資料に限定する。
- 認証はCursorアプリとCursor Agent CLIで別状態になり得る。開始前に `cursor-agent status` でCLI側を確認する。
- 高価なモデルをレビューの必須条件にしない。
- ユーザーが別の担当、モデル、レビュー回数、git操作を明示した場合は、その指示を優先する。
