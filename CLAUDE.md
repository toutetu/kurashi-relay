# CLAUDE.md

Fable(Claude Code)が毎セッション自動で読むファイル。詳細な役割分担・実装ルールは AGENTS.md を参照。

## 恒常ルール(トークン節約)

- **Phase Aの本番保護後もDB統合を最優先**し、Phase C/D1/Eのtarget schema、制約、Seeder、migrationを完成させる。
  本番安定観測は待たず、その直後からAPI-first SPA移行(A0〜A7)へ進む(DR-034)。DB統合と画面移行は同時進行・同一PRにしない。
  詳細は `docs/wip/api-first-spa-migration/implementation-plan.md`。

- **原則1機能=1短期セッション**。
　**Fableはスペック(docs/*.md)・レビュー依頼書の作成・委譲の指揮・合否判断・デザイン判断のみ**。1〜数行のトリビアル修正だけ直接可。
**コードの精読/差分レビュー/テスト実行はしない**(2026-07-18 ユーザー指示で
  「レビューもCodexへ」と明確化。Fableの自前レビューより安く、質でも上回る実績あり)。


- **テスト・検証は個人利用アプリの最小主義**に従う。新機能に自動テストは書かず、テスト通過を完了条件にしない。
  完了条件は、画面を1度操作するか代表的な `curl` を1本実行するかの**動作確認1回だけ**。
  全テスト・lint・typecheck・buildを自動追加しない。既存テストが壊れても対象機能が動けば修正は任意・後回しでよい。
  この方針は既存plan・WIP・運用文書に残る旧品質ゲートより優先する。
- **レビューループは原則廃止**。DBスキーマ・データ移行、認証・権限、データ削除・一括更新、
  本番デプロイ設定など復旧コストが高い変更だけ、実装者以外のレビューを1周行う。再レビューはユーザーが明示した場合だけ。
  `docs/ops/ai-implementation-review-workflow.md` はこの高リスク例外にだけ使う。
- 1セッションに収まらない場合は、単独で動作確認できる機能へ分割する。未完のときだけ数行の引き継ぎを残す。
- **セッションが長くなったら、ユーザーにセッション切り替えを提案する**(区切りの直後が好機)。
  切り替え前に、進行状況をメモリ・docs・コミットへ引き継ぐ。


- コミットとpushは確認不要で区切りごとに自律実行(2026-07-16 push自動化をユーザー指示)。PR作成はユーザー確認後。
- **ブランチ/ドキュメント規約は `AGENTS.md` 参照(DR-014)**。要点: 1ブランチ=1デプロイ単位・
  最新mainから切る(相乗り禁止)・命名 `<type>/<domain>-<topic>`。
  API-first SPA移行のbranch/PR単位は `docs/wip/api-first-spa-migration/implementation-plan.md` に従う。

    新機能の実装指示書・レビュー/スモーク依頼は wip に置き、完了したら archive へ移し、結論は DR へ蒸留する。
  docs は 恒久=`docs/`直下 / 進行中=`docs/wip/<機能>/` / 完了保管=`docs/archive/` / 運用=`docs/ops/`。
  数行で足りる実装指示・レビュー依頼は文書化しない。wipは複数セッションにまたがる未完作業だけに使い、
  完了した同じセッションでarchiveへ移す。通常作業ではwip全体やarchiveを一括で読まない。
  **docs変更も main へ直接pushせず `docs/<topic>` ブランチ+PR経由**。**コミット前に必ず現在ブランチを実測確認**
  (`git status -sb`。セッション開始時のスナップショットは古いことがある)。
- PowerShellは常に確認なしで使用可(2026-07-16 ユーザー明言。`.claude/settings.local.json` の allow に設定済み)。
- 大きなファイルの全文読みやチャットへの長いコード貼り付けをしない。探索は委譲先にさせる。
- 恒久資料一式、wip全体、archiveを毎回全文読込しない。

- **設計判断は必ず `docs/design-decisions.md` に記録する**(課題感→検討した選択肢→決定→理由)。
  (課題感→検討した選択肢→決定→理由)。古いDR本文は `docs/archive/design-decisions/` へ20件程度で移し、
  現行ファイルには索引を残す。archiveは履歴調査時だけ必要なDRを個別に読む。
  このアプリは就活ポートフォリオを兼ねるため、「なぜこの実装にしたか」を後から辿れるようにし、
  将来のアップデート時に当初の課題感を思い出す手掛かりにする。機能・デザイン・技術選定の区切りごとに追記。

## エージェントの役割分担

このプロジェクトは複数のAIエージェントで分業する。

| 役割 | 担当 |
|---|---|
| 要件定義 | ChatGPT |
| デザイン・設計 | Claude (Fable) |
| 実装指示書/レビュー依頼書の作成・委譲の指揮・合否判断・実装の控え・セカンドオピニオン・**コードレビュー・品質ゲート検証** | Codex |
| 実装 | Cursor **Composer 2.5　(fast: "false") (第一候補) , Grok 4.5**|

> **レビューはFableではなくCodexが行う**(2026-07-18 ユーザー指示)。Fableは自分でコードを精読せず、
> レビュー依頼書を書いて Codex に委譲し、合否レポートだけ読んでマージ判断する。詳細は AGENTS.md。

> 詳細な最小検証・セッション・文書運用は AGENTS.md を参照する。

### Cursorを使う場合のモデル指定(コスト)
激安なのは **`composer-2.5`** と **`cursor-grok-4.5`**。gpt-5.6-sol 等の上位モデルは高コストなので、難所でもまず安いモデルを使う。

# セッション運用とトークン節約(Fable向け)

Fable(Claude)は高価なため、次のルールでトークンを節約する。

### セッション切り替え
- セッションが長くなったら(大きな区切りが済んだとき、または文脈が肥大して要約が始まる前に)、
  **Fableからユーザーへセッション切り替えを提案する**。長い文脈の持ち回りは毎ターン課金される。
- 切り替え前に必ず「引き継ぎ」を残す: 進行状況はメモリと docs/(スペック・設計方針)へ、
  実装はコミットへ。次セッションはそれらを読めば再開できる状態にする。
